## 🔴 ROOT CAUSE — fix-grade finding

**File:** `crates/jankurai-runner/src/model_client/runtime.rs`
**Lines:** 159–175 (the `error_text` computation) and 209 (the `success` assignment).

The current logic is:

```rust
let error_text = value
    .as_ref()
    .and_then(|v| v.get("error"))
    .and_then(serde_json::Value::as_str)
    .map(str::to_string)
    .or_else(|| {
        let trimmed = if stderr.trim().is_empty() { stdout.trim() } else { stderr.trim() };
        if trimmed.is_empty() { None } else { Some(trimmed.to_string()) }
    });
// …
let success = output.status.success() && error_text.is_none();
```

**Problem:** the inner `jekko run --ephemeral --json --agent plan <prompt>` subprocess emits **structured-output chatter** on stderr (e.g., `zyalc: wrote …`, tracing init, session creation logs) **even on successful runs**. Because there's no explicit `"error"` field on success, the fallback grabs that stderr and treats it as an error. `error_text.is_some()` ⟹ `receipt.success = false` ⟹ `model_attempt_outcome.state = "model_failure"` ⟹ `ModelOnlyError::ModelFailure` ⟹ recipe exit 1.

The JSON returned by `jekko run` carries an explicit `"success": true` (and `"accepted": true`) field that is being **ignored** by this classifier.

**Reproducer:** any live recipe call where the inner `jekko run` writes anything to stderr but still exits 0 with `success: true` JSON. This will affect P1.b and P1.c too unless the inner-jekko stderr happens to be empty for those code paths.

**Severity:** **CRITICAL** — every live model call is misclassified. The metric `fusion_success_total` increments correctly, but the per-run pipeline halts at the first call.

**Constraint:** `crates/jankurai-runner/src/reasoning_io.rs` (which consumes the receipt) is the 496-LOC score-lever — **do not touch**. The fix lives entirely in `runtime.rs` (314 LOC).

**Proposed minimum diff (FIX-2 candidate):**

Prefer the explicit `success` field in the JSON when present:

```rust
let json_success = value.as_ref().and_then(|v| v.get("success"))
    .and_then(serde_json::Value::as_bool);
let json_error_text = value.as_ref()
    .and_then(|v| v.get("error"))
    .and_then(serde_json::Value::as_str)
    .map(str::to_string);
let error_text = json_error_text.or_else(|| {
    // Only treat stdout/stderr as "error evidence" if the JSON did NOT
    // explicitly confirm success. Successful runs may still write log
    // lines to stderr (zyalc init, tracing config, session boot).
    if json_success == Some(true) { return None; }
    let trimmed = if stderr.trim().is_empty() { stdout.trim() } else { stderr.trim() };
    if trimmed.is_empty() { None } else { Some(trimmed.to_string()) }
});
let success = output.status.success() && error_text.is_none()
    && json_success.unwrap_or(true);
```

This preserves all existing failure paths (timeout, non-zero exit, explicit `"error"` in JSON) but stops misreading successful runs that happen to log to stderr.

I'll land this as **FIX-1** (since it's higher impact than the env-var fix) in Phase 4.
