## 🔴 Runner-side mis-classification — same fix-grade bug as P1.a

Every attempt returned `success: false`. The recipe stderr proves what actually happened:

```json
{
  "assistant_text": "I cannot complete this request because the necessary evidence and research receipts are not available.",
  "provider_id": "jnoccio", "model_id": "jnoccio-router",
  "credential_source_policy": "users-only", "selected_credential_user_id": "user_1",
  "credential_user_id": "user_1", "accepted": true, "success": true
}
```

The model is **refusing the task** (legitimate content response — it's asked to do a literature synthesis but evidence array is `[]`). But because the runner's `model_client/runtime.rs` misclassifies the call as failed (stderr non-empty heuristic), the hero-judge classifier in `hero_judge_runner_completion.rs:199` short-circuits at `if !receipt.success` and never even reads `assistant_text`.

### After the FIX-1 patch (proposed in P1.a report):

- `receipt.success` would correctly be `true`.
- `classify_hero_completion` would reach line 218: `let Some(text) = receipt.response.as_deref()` ✓
- `parse_structured_model_json(text)` would fail because the response is plain English, not JSON.
- With `require_parsed_live_json: true`, line 230-237 would emit `RetryableFailure("live model response was not parseable JSON")` (attempts 1,2) and `FinalBlock` (attempt 3).
- The run would still fail — but with the **right reason** (content issue, model declined the task) rather than the wrong reason (false alarm on model success).

That's the correct semantics: distinguish "model declined / can't comply" from "model never responded".
