## Executive recommendation

Yes: the current system is model-picker-first. It should become **key-file-first + automatic routing**.

The clean target is:

1. **Users drop keys into one central secrets file.**
2. **Jekko discovers providers automatically from known key names.**
3. **Jekko routes automatically by task, capabilities, context, cost, and token estimate.**
4. **Free users get one active model key/credential.** Extra keys can exist in the file, but they are ignored for routing and clearly shown as inactive.
5. **Developer-key users get multi-key / multi-provider routing.**
6. **Full unlimited paid Jnoccio routing stays in protected code**, loaded only when the developer entitlement is valid.
7. **Paid models are opt-in by model name** in a separate non-secret config file, so users do not accidentally spend money.

The uploaded codebase already has most provider metadata needed, but the selection flow is spread across config, provider runtime, TUI state, ACP, session creation, and CLI commands. The work is not just adding a router; it is removing model selection as a normal user concept.

---

## Current-state diagnosis

The current architecture lets users select models in many places:

* `MODEL_SELECTION_ARCHITECTURE.md` documents model selection through the TUI, slash commands, CLI model lists, session overrides, and config.
* `packages/jekko/src/config/config-schema.ts` exposes top-level `model` and `small_model`.
* `packages/jekko/src/provider/provider-runtime.ts` resolves a default model from `cfg.model`, recent model state, then provider/model sorting.
* `packages/jekko/src/cli/cmd/tui/context/local-model.ts` stores selected model, recent models, favorites, variants, and per-agent overrides.
* `packages/jekko/src/cli/cmd/tui/component/dialog-model.tsx` is a full TUI model picker.
* `packages/jekko/src/cli/cmd/tui/app-commands-agent.tsx` exposes “Switch model” and hidden cycle-model commands.
* `packages/jekko/src/session/prompt.ts` stores and reuses a concrete model on user messages.
* `packages/jekko/src/acp/agent.ts` exposes model selection/config options to ACP clients.

So the current UX is fundamentally: **pick provider/model, then run**.

The target UX should be: **add keys, optionally enable paid models, then Jekko chooses**.

## Provider key conventions to support

The central key file should normalize aliases rather than assume only one env var per provider. The important ones are:

```env
OPENAI_API_KEY=...
ANTHROPIC_API_KEY=...
GEMINI_API_KEY=...
GOOGLE_API_KEY=...
GOOGLE_GENERATIVE_AI_API_KEY=...
JNOCCIO_DEVELOPER_KEY=...
```

OpenAI’s official Node SDK defaults to `OPENAI_API_KEY`; Anthropic documents `ANTHROPIC_API_KEY`; Google Gemini documents `GEMINI_API_KEY` and `GOOGLE_API_KEY`, while the Vercel AI SDK Google provider uses `GOOGLE_GENERATIVE_AI_API_KEY`. Jekko should accept all of these and normalize them internally. ([GitHub][1])

## Recommended new user-facing setup

### 1. Central secrets file

Use a single `.env`-style secrets file:

```bash
~/.config/jekko/keys.env
```

Example:

```env
# ~/.config/jekko/keys.env
# chmod 0600

OPENAI_API_KEY=sk-...
ANTHROPIC_API_KEY=sk-ant-...
GEMINI_API_KEY=...
GOOGLE_GENERATIVE_AI_API_KEY=...

# Unlocks protected Jnoccio developer routing.
JNOCCIO_DEVELOPER_KEY=...
```

Why `.env` instead of JSON:

* The user explicitly wants expected ENV names.
* It is copy/paste friendly.
* It avoids deep provider config.
* It is compatible with existing CLI/server deployment habits.
* It can be loaded without requiring shell export.

Do **not** make this project-discoverable by default. Secrets should live globally, not in `.jekko/keys.env` inside repos. Add an explicit override for automation:

```bash
JEKKO_KEYS_FILE=/path/to/keys.env
JEKKO_KEYS_CONTENT="OPENAI_API_KEY=..."
```

### 2. Separate non-secret routing/model policy file

Use a second config file for paid-model opt-in:

```bash
~/.config/jekko/router.jsonc
```

Example:

```jsonc
{
  "$schema": "https://jekko.ai/router.schema.json",

  // Free / zero-cost models are eligible automatically.
  // Paid models must be explicitly listed here.
  "allowed_paid_models": [
    "openai/gpt-5-mini",
    "anthropic/claude-haiku-4-5",
    "google/gemini-2.5-flash",
    "codex/gpt-5.3-codex"
  ],

  "blocked_models": [],

  "preferences": {
    "goal": "minimize_cost",
    "free_first": true,
    "max_output_tokens": "auto"
  }
}
```

This keeps secrets and policy separate:

* `keys.env` = credentials.
* `router.jsonc` = what paid models the user consents to use.

This also solves the “some people will only put in a paid API key” problem without accidental spend. If they only add a paid key and no paid model allow-list, Jekko should say:

> Key found. No paid models are enabled. Add a model to `allowed_paid_models` to allow paid usage.

The CLI can provide a helper command that appends the cheapest recommended model for a provider, but the actual consent should still be represented as a model name in config.

## Critical design warning: do not treat `cost: 0` as automatically free

The existing code defaults missing cost fields to zero in model normalization. That is dangerous for a cost router.

A missing cost can currently look like a free model.

Recommended fix:

```ts
type PricingKind = "free" | "paid" | "unknown"

interface RoutedModelMetadata {
  pricingKind: PricingKind
  costKnown: boolean
  cost: Provider.Cost
}
```

Router policy should be:

* `free`: automatically eligible.
* `paid`: eligible only if listed in `allowed_paid_models`.
* `unknown`: not eligible by default unless explicitly allow-listed, and the UI should label it as “cost unknown.”

This is one of the most important changes in the project because otherwise Jekko could accidentally route paid traffic through a model whose pricing metadata was simply absent.

```ts
interface ProviderCredential {
```

```ts
interface RoutingDecision {
  selected: {

    paidAllowed: boolean
    protectedRouterUsed: boolean

  estimate: {
    inputTokens: number
    outputTokens: number
    estimatedCostUSD?: number
    costKnown: boolean

  reason: string

  candidates: Array<{
    fullID: string
    eligible: boolean
    rejectedReason?: string
    estimatedCostUSD?: number
    pricingKind: "free" | "paid" | "unknown"

}
```

#### `router/public-router.ts`

This is the public/free implementation.

It should:

* Route automatically.
* Prefer eligible free models.
* Use only one active credential in free mode.

* Provide clear explanations.

It should **not** contain the full proprietary unlimited paid balancing logic.

#### `router/protected-router-loader.ts`

This is where the protected Jnoccio router is loaded.

Example:

```ts
export async function loadProtectedRouter(entitlement: Entitlement) {

  try {
    return await import("@jekko-ai/jnoccio-router-protected")
  } catch {
    return undefined
  }
}
```

The public repository should only include the interface and fallback behavior. The full multi-key paid balancing logic should live in protected code.

## Free vs developer behavior

### Free version

Free users can have a `keys.env` like this:

```env
OPENAI_API_KEY=...
ANTHROPIC_API_KEY=...
GEMINI_API_KEY=...
```

But Jekko should activate only **one** model key/credential.

Recommended free-tier policy:

1. Load all keys.
2. Show all keys in status, redacted.
3. Pick one active credential deterministically.
4. Mark all other credentials inactive with reason:

```text
inactive: free tier supports one active model key
```

Recommended active-key selection order:

1. Jekko/Jnoccio free credential, if present.
2. Provider with eligible free models.
3. Provider with an explicitly allow-listed paid model and lowest estimated cost.
4. Otherwise, no route; show setup instructions.

The important thing: free users should not get multi-provider smart routing just because they pasted several keys.

### Developer-key version

Developer key unlocks:

* Multiple active credentials.
* Multi-provider paid routing.
* Fallback across providers.
* Load-aware balancing.
* Rate-limit-aware retries.
* Protected Jnoccio optimization.

The public code should still enforce:

* Paid models only from `allowed_paid_models`.
* Keys redacted.
* Unknown-cost warning.
* Protected router unavailable fallback.

## Routing algorithm

The router should decide at request time.

### Inputs

```ts
interface RoutingInput {
  taskType:
    | "chat"
    | "title"
    | "summary"
    | "tool"
    | "shell"
    | "subtask"
    | "vision"
    | "code"
    | "reasoning"

  required: {
    text: boolean
    toolCall: boolean
    imageInput: boolean
    pdfInput: boolean
    structuredOutput: boolean
    reasoning: boolean
    minContextTokens: number
    minOutputTokens: number
  }

  estimatedTokens: {
    input: number
    output: number
    cacheRead?: number
    cacheWrite?: number
  }

  sessionID?: string
  agentName?: string
  previousFailures?: Array<{
    providerID: string
    modelID: string
    reason: "auth" | "rate_limit" | "timeout" | "server_error"
  }>
}
```

### Candidate filtering

A model is eligible only if:

1. Provider has an active credential.
2. Model is not locked.
3. Model is active.
4. Model supports required capabilities.
5. Context/output limits are adequate.
6. Model is not blocked.
7. Pricing policy allows it:

   * free: eligible by default;
   * paid: eligible only if listed in `allowed_paid_models`;
   * unknown: ineligible unless explicitly listed.

### Scoring

Primary objective:

```text
lowest expected total cost for the required capability
```

Estimated cost:

```text
inputTokens * inputCost
+ outputTokens * outputCost
+ cacheReadTokens * cacheReadCost
+ cacheWriteTokens * cacheWriteCost
```

Add special handling for over-200k pricing, because the existing provider schema already has `experimentalOver200K` cost fields.

Tie-breakers:

1. Free beats paid when capability is adequate.
2. Lower estimated cost.
3. Smaller sufficient model beats larger model.
4. Better context fit.
5. Better task fit.
6. Reliability / recent failure avoidance.
7. Stable deterministic ordering.

### Token optimization

Model selection alone does not minimize token use. Jekko should also route with token policy:

* Use small/cheap models for title, summary, shell command naming, and short subtasks.
* Use higher-context models only when the prompt requires it.
* Auto-cap output tokens by task type instead of always exposing max output.
* Trigger summarization/compaction before routing if the prompt would force an expensive large-context model.
* Prefer cached-token economics where supported.
* Store actual token usage after completion and feed that into later estimates.

The existing hardcoded `getSmallModel()` logic in `provider-runtime.ts` should become a router intent:

```ts
taskType: "title" | "summary" | "subtask"
```

Not a separate hardcoded model list.

Add:

```bash

```

5. Leave OAuth credentials untouched.
6. Warn but not destroy old files automatically.

Example output:

```text

  config.model anthropic/claude-sonnet-4 -> router.jsonc allowed_paid_models

Run jekko keys doctor for cleanup guidance.
```

## Recommended product behavior matrix

| User state                                 | Behavior                                                                             |
| ------------------------------------------ | ------------------------------------------------------------------------------------ |
| No keys                                    | Jekko shows keys setup path and uses built-in/free Jekko provider only if available. |
| One free/zero-cost API key                 | Auto-route to best eligible free model.                                              |
| One paid API key, no paid model allow-list | Do not spend. Show “paid key found, no paid models enabled.”                         |
| One paid API key, paid model allow-listed  | Auto-route within that provider/model set.                                           |
| Multiple API keys, no developer key        | Only one credential active. Others shown inactive.                                   |
| Multiple API keys, developer key           | Protected Jnoccio router can balance across all enabled providers/models.            |
| Unknown-cost model                         | Block unless explicitly allow-listed; show cost unknown.                             |
| Codex OAuth connected                      | Candidate only if auth exists and policy allows the model.                           |

## Final recommendation

Build this around three concepts:

```text
keys.env        → what credentials exist
router.jsonc    → what paid models are allowed
ModelRouter     → which model should be used now
```

Then remove the normal user-facing model picker.

The protected boundary should be:

```text
public code:
  key loading
  entitlement check
  one-key/free routing
  paid allow-list enforcement
  basic cost scoring

protected Jnoccio code:
  unlimited paid balancing
  advanced multi-provider optimization
  proprietary ranking/fallback logic
```

That gives you the setup you want:

```text
User drops keys in.
Jekko detects providers.
Jekko chooses the best free/cheap eligible model.
Paid usage requires explicit model allow-list.
Developer key unlocks full Jnoccio routing.
```

And it avoids the current anti-pattern where users are pushed into thinking about providers, model IDs, favorites, recents, and manual switching.
