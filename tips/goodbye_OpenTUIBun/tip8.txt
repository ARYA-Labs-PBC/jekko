For **only Rust TUI, never web**, my strongest recommendation is:

**Use Ratatui + Crossterm as the main UI stack. Do not switch to OpenTUI. Remove Bun. Strongly consider removing Vite from the production app too.**

The clean version:

```text
Rust TUI app:
  Rust
  Cargo
  Ratatui
  Crossterm
  Tokio if you need async/background work
  Tree-sitter / syntect / similar only if you need code rendering
  No Bun
  No Vite
  No TypeScript in the runtime path
```

OpenTUI is cool, but for your stated target it pulls you in the wrong direction. Its docs describe it as a native terminal UI core written in Zig with TypeScript bindings, and its getting-started page currently says OpenTUI is **Bun-exclusive** while Deno/Node support is still in progress. ([OpenTUI][1]) That means using OpenTUI today effectively adds **Bun + TypeScript + Zig-native dependency surface** to a project that you want to be Rust-native.

Ratatui, by contrast, is directly a **Rust library for building fast, lightweight, rich terminal user interfaces**. ([Ratatui][2]) Its docs describe it as a lightweight Rust crate with widgets and utilities for complex Rust TUIs, usually paired with `crossterm`. ([Docs.rs][3]) That is exactly your target.

## My blunt opinion

You should **move away from OpenTUI for this project**, not away from Ratatui.

OpenTUI makes sense when the app itself is TypeScript/Bun-first and you want React/Solid-style terminal components. But you are saying **only Rust TUI, never web**. In that world, OpenTUI is not simplifying anything. It is adding a second ecosystem.

## What Bun is

**Bun** is an all-in-one JavaScript/TypeScript tool. It can run JS/TS files, install packages, run tests, and bundle code. The Bun docs describe it as a single executable with a runtime, package manager, test runner, and bundler. ([Bun][4])

So in practical terms, Bun is like some combination of:

```text
node
npm / pnpm / yarn
tsx / ts-node
jest-ish test runner
bundler
```

For OpenTUI, Bun matters because the official OpenTUI getting-started flow is:

```bash
bun init -y
bun add @opentui/core
bun index.ts
```

That is not a Rust-native path. ([OpenTUI][1])

## Does removing Bun simplify your stack?

**Yes. Absolutely.**

For your project, removing Bun simplifies the stack because it removes:

```text
one JavaScript runtime
one package manager path
one lockfile/tooling ecosystem
one extra native dependency path
one source of CI/build complexity
one reason your TUI is not purely Rust-native
```

But there is an important catch: **if you keep Vite, you still need a JavaScript runtime/package manager somewhere.**

Vite is officially described as “The Build Tool for the Web” and as frontend tooling for web applications. ([vitejs][5]) Its guide describes it as tooling for modern web projects, including a dev server and HMR. ([vitejs][6])

So if you are truly doing **no web**, then Vite is probably also unnecessary for the actual product.

## Your stack as stated: Rust / Vite / TypeScript, but no web

That stack is internally awkward.

**Rust + Ratatui** makes sense for a TUI.

**Vite + TypeScript** makes sense for a browser UI, web dashboard, docs site, report viewer, or maybe a design/prototyping layer.

But **Vite + TypeScript + no web + Rust TUI** is not a natural production stack. It can work, but it creates extra moving parts without helping the terminal app much.

I would restructure it like this:

```text
Production app:
  Rust only

Terminal UI:
  ratatui
  crossterm
  tokio, only where async actually helps

Testing:
  cargo test
  cargo nextest
  insta for snapshots
  ratatui TestBackend for frame snapshots
  portable-pty / vt100 if you want black-box TUI testing

TypeScript:
  optional only for dev scripts, docs, generators, or a report viewer

Vite:
  optional only if you have an actual browser-based report viewer/docs/demo

Bun:
  remove, unless you deliberately choose OpenTUI
```

## What I would choose for you

Given your repeated goal of a **sharp Rust-native terminal UI**, I would use:

```toml
[dependencies]
ratatui = "0.30"
```

## When I would keep TypeScript

I would keep TypeScript only for things like:

```text
generated docs
configuration schema tooling
developer dashboard
test report viewer
static site
design/prototype sandbox
API contract generation
```

I would **not** let TypeScript own the TUI runtime.

That means this is okay:

```text
core app: Rust
TUI: Rust/Ratatui
optional docs/report viewer: TypeScript/Vite
```

This is what I would avoid:

```text
core app: Rust
TUI shell: TypeScript/OpenTUI/Bun
renderer: Zig/OpenTUI
build: Vite/Bun
```

That second version is cool, but it fights your “only Rust TUI” goal.

## Final recommendation

Stay on **Ratatui**.

Use **Crossterm** under it.

Remove **Bun** unless you intentionally adopt OpenTUI.

Remove **Vite** from the production TUI path. Keep it only for a web-based docs/reporting sidecar.

Treat **TypeScript** as optional tooling, not as the terminal app’s UI layer.

My ideal answer for your project would be:

```text
Rust app.
Ratatui UI.
Crossterm terminal backend.
Cargo build.
Cargo test.
No Bun.
No OpenTUI.
No Vite unless there is an actual browser artifact.
```

So yes: **removing Bun simplifies your stack**, and for a Rust-only TUI, **Ratatui is the better long-term bet.**
