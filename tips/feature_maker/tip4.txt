Yes. The feature-upgrade agent should be a **strategic capability agent**, not just a “semantic improvement” agent. Your bug-finder baseline already has the right discipline: repo map → hypothesis → invariant → counterexample → proof. For feature discovery, the analogous discipline is: **mission map → capability graph → unmet user/job gap → feature hypothesis → implementation plan → measurable proof**. The prior bug-finder design explicitly emphasized cross-file reasoning, contracts, tests, data models, deployment config, and evidence-backed PRs; this new system should keep that rigor but aim at capability expansion instead of invariant failure.  

# 1. Core framing

The agent should not ask:

> “What feature could we add?”

It should ask:

> “What capability does this project appear to be trying to become, what is missing from that capability, and can I propose a concrete upgrade with implementation path, verification plan, and strategic justification?”

That means the agent needs to model five things at once:

```text
1. Product intent
   What is the project for? Who is it for? What future does it imply?

2. Core IP
   What is the unique engine, workflow, data model, algorithm, integration,
   ontology, UX loop, or domain-specific trick that makes this repo special?

3. Current capabilities
   What can the product actually do today, as proven by code, APIs, UI flows,
   database schema, tests, docs, and runtime behavior?

4. Capability gaps
   What important adjacent capability is implied by the mission but absent,
   weak, hidden, unproductized, or blocked by architecture?

5. Implementable upgrade path
   What is the smallest serious feature slice that upgrades the product,
   preserves architecture, and can be verified end-to-end?
```

For bugs, the forcing function is **counterexample**. For features, the forcing function should be **capability proof**:

```text
A proposed feature is not valid until the agent can show:
- why the project wants it,
- which existing capability it extends,
- which users/workflows benefit,
- which code/API/data surfaces it touches,
- how to implement it incrementally,
- how to verify it through tests, UX flows, contracts, metrics, and rollout.
```

# 3. The “context smart, core inside out” model

The agent should not ingest the repo as a flat bag of chunks. That is the main failure mode. It should build a **project cognition graph** from the inside out.

Use this context onion:

```text
Layer 0: Product thesis
What is this project trying to become?

Layer 1: Core IP
What is the unique engine, algorithm, domain model, workflow, or data asset?

Layer 2: Capabilities
What user-visible and API-visible abilities exist today?

Layer 3: Workflows
How do users move through the product? What states and actions exist?

Layer 4: Contracts
What APIs, schemas, generated types, and validation rules define interaction?

Layer 5: Data model
What entities, relationships, constraints, and state machines exist?

Layer 6: Implementation
Which modules, functions, components, services, and jobs implement the above?

Layer 7: Runtime reality
What Docker/CI/deployment/env assumptions constrain the feature?

Layer 8: Proof
What tests, benchmarks, traces, contracts, and analytics prove behavior?
```

The agent should move in both directions:

```text
inside → out:
mission → core IP → capability → workflow → API/UI → code → tests

outside → in:
route/API/table/component → capability → user job → mission/IP relevance
```

That two-way traversal is what prevents shallow suggestions like “add dark mode” when the real upgrade is “turn the internal planning engine into a reusable API with an evaluation harness.”

# 4. Recommended architecture

I would build this as a multi-agent system, but with one shared evidence store. The agents should not freestyle independently. They should write facts into a typed project graph and retrieve from that graph.

```text
┌─────────────────────────────────────────────────────────────┐
│                   Strategic Feature Agent                    │
├─────────────────────────────────────────────────────────────┤
│  1. Repo Scout                                               │
│  2. Mission / Product Synthesizer                            │
│  3. Code Intelligence Builder                                │
│  4. API + Workflow Mapper                                    │
│  5. Data Model + State Machine Mapper                        │
│  6. Runtime / Deployment Mapper                              │
│  7. Core IP Detector                                         │
│  8. Capability Gap Generator                                 │
│  9. Feature Scorer / Critic                                  │
│ 10. Implementation Planner                                   │
│ 11. Verification Planner                                     │
│ 12. PR / RFC Writer                                          │
└─────────────────────────────────────────────────────────────┘
```

## 4.1 Repo Scout

Purpose:

```text
- Find canonical files.
- Identify stack.
- Identify package manager.
- Identify build/test commands.
- Identify service boundaries.
- Identify frontend/backend/db/deploy structure.
- Produce a first-pass repo map.
```

Outputs:

```yaml
repo:
  stack:
    backend: rust
    frontend: react-vite-typescript
    database: postgres_or_sqlite_or_mysql
    deployment: docker
  package_manager: pnpm|npm|yarn
  rust_workspace: true|false
  api_specs:
    - openapi.yaml
  db_migrations:
    - migrations/
  frontend_routes:
    - src/routes/
  test_commands:
    - cargo test
    - pnpm test
    - pnpm build
```

## 4.2 Mission / Product Synthesizer

Purpose:

```text
- Read README, docs, roadmap, issues, PR templates, examples.
- Extract project nouns, verbs, personas, promised outcomes, non-goals.
- Infer product thesis and maturity stage.
```

Output example:

```yaml
product_thesis:
  one_sentence: >
    This project helps teams convert domain-specific workflows into auditable,
    reproducible planning runs.
  target_users:
    - internal operators
    - developers
    - admins
  core_jobs:
    - configure workflow
    - run workflow
    - inspect result
    - compare runs
    - export/share evidence
  implied_future:
    - collaboration
    - versioning
    - evaluation
    - API access
```

## 4.3 Code Intelligence Builder

Use multiple indexes, because no single index is enough.

```text
A. Lexical index
   Exact search over filenames, symbols, comments, docs, SQL, config.

B. AST index
   Structural code facts from parsers.

C. Symbol graph
   Definitions, references, imports, call graph, type graph.

D. Semantic/vector index
   Meaning-based retrieval over docs, code comments, tests, markdown, issues.

E. Runtime trace index
   What actually happens when users click, APIs run, and DB queries execute.

F. Git/change index
   What changed recently, which files change together, what is stale.
```

Tree-sitter is a strong fit for structural parsing because it is both a parser generator and an incremental parsing library that builds concrete syntax trees for source files. ([Tree-sitter][4]) LSP should also be used because it exposes IDE-grade features such as autocomplete, go-to-definition, and find-references through a standard protocol. ([Microsoft GitHub][5]) For persistent precise code intelligence, SCIP is useful because it is a language-agnostic indexing protocol for code navigation such as go-to-definition and find-references. ([SCIP Code][6])

## 4.4 API + Workflow Mapper

Purpose:

```text
- Map frontend routes to components.
- Map components to API calls.
- Map API calls to backend handlers.
- Map handlers to services/repositories.
- Map repositories to SQL tables.
- Map all of that to tests.
```

Example graph path:

```text
Page:/projects/:id
  renders Component:ProjectDashboard
  calls GET /api/projects/:id/summary
  handled_by RustFunction:get_project_summary
  reads Table:projects
  reads Table:runs
  reads Table:metrics
  tested_by e2e/project_dashboard.spec.ts
```

This is where many feature opportunities appear:

```text
- API exists but no UI.
- UI exists but no persistence.
- Table supports history but app exposes only current state.
- Backend can compare things but frontend cannot visualize comparison.
- Tests show a workflow but docs never mention it.
- Docs promise a workflow but implementation is incomplete.
```

## 4.5 Data Model + State Machine Mapper

Purpose:

```text
- Extract tables, columns, enums, foreign keys, indexes.
- Identify state machines from enum/status columns.
- Map business objects to API DTOs and UI models.
- Find latent feature affordances.
```

Latent affordance examples:

```text
has created_at/updated_at but no history view
has status enum but no workflow visualization
has organization_id but no team management UI
has run_id/result_id but no comparison UI
has metadata JSON but no schema explorer
has audit table but no audit UI
has API keys table but no developer portal
has export table but no export workflow
```

## 4.6 Runtime / Deployment Mapper

Purpose:

```text
- Understand whether the project can support the proposed feature operationally.
- Detect existing queues, workers, cron jobs, websockets, object storage, cache,
  email, auth, background processing, or observability.
```

Feature feasibility depends heavily on runtime shape:

```text
Feature needs background jobs?
  Check for worker process, queue, cron, Docker Compose services.

Feature needs live collaboration?
  Check websocket/SSE support, auth sessions, data conflict model.

Feature needs import/export?
  Check object storage, file upload limits, MIME validation, async processing.

Feature needs auditability?
  Check event log, actor_id columns, trace IDs, immutable history.
```

## 4.7 Core IP Detector

This is one of the most important agents.

It should identify the project’s **defensible center**:

```text
- domain-specific algorithm
- planner/compiler/evaluator
- workflow engine
- data model or ontology
- scoring/ranking system
- simulation engine
- rules engine
- integration layer
- proprietary templates
- evaluation harness
- dataset or feedback loop
- unique UX workflow
- API protocol
```

Signals for core IP:

```text
- central module with high fan-in/fan-out
- docs repeatedly refer to it
- tests are deep around it
- examples showcase it
- CLI exposes it
- API wraps it
- database records its outputs
- users would not use the product without it
- it is hard to replace with a generic library
```

The agent should produce:

```yaml
core_ip_candidates:
  - name: "planning engine"
    evidence:
      - crates/planner/src/lib.rs
      - docs/MISSION.md
      - examples/basic_plan.rs
      - tests/planner_regression.rs
    why_it_matters: "All product workflows converge on generating and evaluating plans."
    upgrade_vectors:
      - versioned plans
      - explainability
      - comparison
      - benchmark suite
      - API/SDK
      - plugin system
```

# 5. Technology stack I would use

## Retrieval and indexing

Do **not** choose between graph DB, vector DB, and RAG. Use all of them, but for different jobs.

```text
BM25 / lexical search:
  Best for exact names, symbols, filenames, env vars, SQL table names.

Vector search:
  Best for fuzzy concepts, mission docs, similar workflows, semantic gaps.

Graph DB:
  Best for relationships: route → API → handler → query → table → test.

AST / LSP / SCIP:
  Best for precise code navigation and symbol-level reasoning.

Runtime traces:
  Best for actual behavior, not just static structure.

Summaries:
  Best for compressing large areas, but only after facts are grounded.
```

Qdrant is a good example of a vector search engine designed for semantic search and vector similarity retrieval. ([Qdrant][7]) Neo4j’s property graph model is a good conceptual match for this because it represents entities as nodes connected by relationships. ([Graph Database & Analytics][8]) GraphRAG-style retrieval is relevant because it combines graph extraction, hierarchy/community summarization, and retrieval over structured relationships instead of relying only on plain text snippets. ([Microsoft GitHub][9])

## Practical storage choices

For an MVP:

```text
SQLite/Postgres:
  facts, files, symbols, summaries, scores

pgvector or Qdrant:
  embeddings

Tantivy/OpenSearch/Elasticsearch:
  lexical search

NetworkX/Kuzu/Neo4j:
  graph queries

Local JSONL artifacts:
  cheap intermediate evidence logs
```

For a serious production system:

```text
Postgres:
  canonical facts, runs, evidence, agent outputs, scoring history

Qdrant:
  semantic retrieval

OpenSearch/Elasticsearch/Tantivy:
  exact text and hybrid search

Neo4j / Memgraph / Kuzu:
  code-product-runtime graph

Object storage:
  screenshots, traces, generated reports, SARIF, coverage, benchmark outputs

Temporal / durable workflow engine:
  long-running ingestion, indexing, evaluation, PR generation

OpenTelemetry:
  runtime traces, metrics, logs, correlation IDs
```

## Static analysis and code-as-data

CodeQL should be considered for deeper semantic queries because it treats code as data and lets you write queries over a codebase. ([CodeQL][10]) But CodeQL is not a replacement for the feature agent. It is a supporting evidence engine.

For this stack, I would use:

```text
Rust:
  cargo metadata
  rust-analyzer / LSP
  cargo test
  cargo clippy
  cargo doc
  cargo llvm-cov
  cargo expand where useful
  sqlx metadata if SQLx is used

TypeScript/React/Vite:
  tsserver / LSP
  tsc --noEmit
  eslint
  vite build
  vitest
  Playwright
  dependency graph from package manager

SQL:
  migration parser
  schema introspection
  EXPLAIN plans for performance-sensitive features
  migration apply/rollback tests
  constraint tests

Docker:
  docker build
  docker compose
  image inspection
  healthcheck smoke tests
```

# 8. Feature discovery modes

I would run these as rotating modes.

## 8.1 Mission gap hunter

Goal:

```text
Find what the docs imply but the product does not yet do.
```

Questions:

```text
What does the README promise?
What does the roadmap hint at?
What is described as important but not represented in API/UI/data/tests?
What examples feel more powerful than the actual product?
What “coming soon” or “future work” item is now technically easy?
```

Good outputs:

```text
Feature: turn internal run output into shareable reports.
Feature: add first-class project templates because docs already show repeated setup.
Feature: add comparison mode because evaluation is central but only single-run detail exists.
```

## 8.2 Latent capability hunter

Goal:

```text
Find capabilities already half-built in code/data but not productized.
```

Signals:

```text
unused table
unused enum variant
internal endpoint
CLI-only command
feature flag
hidden component
unlinked route
test-only helper
manual script
admin-only workflow
```

This is usually the highest ROI feature mode because implementation cost is lower.

## 8.3 Workflow completion hunter

Goal:

```text
Find workflows with missing beginning, middle, or end.
```

Examples:

```text
Can create but not duplicate.
Can run but not compare.
Can import but not validate.
Can view but not export.
Can configure but not test config.
Can invite but not manage roles.
Can generate but not explain.
Can fail but not recover.
Can search but not save search.
```

## 8.4 API productization hunter

Goal:

```text
Turn internal capability into external platform capability.
```

Upgrade paths:

```text
internal handler → public API
API → OpenAPI spec
OpenAPI spec → generated SDK
SDK → examples
examples → developer portal
developer portal → API keys/rate limits
```

## 8.5 Data/IP upgrade hunter

Goal:

```text
Strengthen the project’s core defensibility.
```

Feature types:

```text
evaluation harness
benchmark corpus
versioned outputs
explainability layer
traceable decisions
domain ontology
feedback loop
plugin system
custom scoring engine
workflow replay
simulation mode
```

This is where “core IP” matters most. The agent should prefer upgrades that make the unique engine more powerful, measurable, reusable, or defensible.

## 8.6 Enterprise readiness hunter

Goal:

```text
Find the feature that makes adoption by serious teams possible.
```

Common upgrades:

```text
SSO/SAML/OIDC
audit logs
role-based access control
team/org management
API keys
usage limits
billing hooks
data export
retention controls
admin dashboard
deployment hardening
```

This is not always glamorous, but it often unlocks actual usage.

## 8.7 Integration hunter

Goal:

```text
Find adjacent tools the project should connect to.
```

Signals:

```text
manual copy/paste workflow
exported JSON/CSV used elsewhere
docs mention another tool
users likely arrive from GitHub/Slack/Jira/Linear/Notion/etc.
project has webhooks but no integration UI
```

## 8.8 Developer-experience hunter

Goal:

```text
Find improvements that make the project easier to extend.
```

Examples:

```text
plugin API
SDK
CLI
typed client generation
local dev bootstrap
example gallery
test harness
schema explorer
mock server
```

# 9. Feature scoring rubric

The agent should not merely rank by excitement. It should score using evidence.

I would use two separate scores:

```text
Opportunity Score:
  Is this feature worth doing?

Execution Readiness Score:
  Can this feature be done safely and incrementally now?
```

## Opportunity Score

```yaml
mission_fit: 0-10
user_value: 0-10
core_ip_leverage: 0-10
capability_gap_strength: 0-10
market_or_adoption_relevance: 0-10
strategic_compounding: 0-10
```

Meaning:

```text
mission_fit:
  Does this directly advance the product thesis?

user_value:
  Does it solve a real workflow pain or unlock a valuable job?

core_ip_leverage:
  Does it strengthen the project’s unique engine/workflow/data advantage?

capability_gap_strength:
  Is the gap clearly evidenced by docs/code/API/UI/data?

market_or_adoption_relevance:
  Would users, teams, or developers plausibly care now?

strategic_compounding:
  Does it enable future features, ecosystem, data, or monetization?
```

## Execution Readiness Score

```yaml
implementation_clarity: 0-10
architecture_fit: 0-10
testability: 0-10
runtime_feasibility: 0-10
risk_containment: 0-10
incremental_sliceability: 0-10
```

Meaning:

```text
implementation_clarity:
  Are the touched files, APIs, DB changes, and UI changes known?

architecture_fit:
  Does it extend existing patterns instead of forcing a rewrite?

testability:
  Can it be proven with unit, integration, e2e, contract, migration, or benchmark tests?

runtime_feasibility:
  Does current Docker/CI/deployment support it?

risk_containment:
  Can it avoid high-risk auth/security/data-loss surfaces or handle them carefully?

incremental_sliceability:
  Can it ship as a vertical slice rather than a giant platform rewrite?
```

## Combined formula

A useful formula:

```text
Feature Priority =
  (Opportunity Score × Confidence × Strategic Multiplier)
  / (Effort × Risk × Uncertainty)
```

Where confidence is evidence-based:

```yaml
confidence:
  docs_support: 0-1
  code_support: 0-1
  api_support: 0-1
  data_model_support: 0-1
  workflow_support: 0-1
  test_support: 0-1
  runtime_support: 0-1
```

Penalties:

```text
- speculation penalty:
    high if the feature is based on vague vibes rather than repo evidence

- rewrite penalty:
    high if the feature requires replacing core architecture

- orphan penalty:
    high if it does not connect to mission/core IP

- proof penalty:
    high if success cannot be tested or measured

- adoption penalty:
    high if it adds capability but no discoverable workflow
```

Every proposed feature should be emitted as a card like this:

# 11. Verification strategy for feature upgrades

A feature PR should not be accepted because the code compiles. It should prove that a capability was added.

## Proof levels

```text
Level 1: Unit proof
  Core transformation, validation, scoring, parsing, formatting.

Level 2: Contract proof
  API request/response schema, generated clients, error cases.

Level 3: Data proof
  Migration, constraints, nullability, rollback/forward behavior.

Level 4: Workflow proof
  User can complete the workflow in browser automation.

Level 5: Runtime proof
  Docker Compose app boots and feature works in realistic environment.

Level 6: Observability proof
  Feature emits useful logs/traces/metrics/events.

Level 7: Product proof
  The feature maps to a user job and success metric.
```

# 13. Best feature-upgrade categories

These are the categories I would use in recurring runs.

| Category             | What it finds                                     | Example                                        |
| -------------------- | ------------------------------------------------- | ---------------------------------------------- |
| Mission gap          | Docs/roadmap imply a missing feature              | README says “evaluate,” but no comparison UI   |
| Latent capability    | Code/data already half-supports something         | Audit table exists but no audit viewer         |
| Workflow completion  | User can start but not finish a job               | Can import data but cannot validate/fix errors |
| API productization   | Internal capability could become platform feature | Add OpenAPI + SDK + API keys                   |
| Core IP upgrade      | Strengthens the project’s unique engine           | Add benchmark/evaluation harness               |
| Data-model unlock    | Schema supports richer product behavior           | Versioned records become history timeline      |
| Enterprise readiness | Makes serious adoption possible                   | RBAC, audit logs, SSO, retention               |
| Integration          | Connects product to adjacent workflow             | GitHub/Slack/Jira/Linear export/import         |
| Developer experience | Makes repo extensible                             | Plugin system, CLI, generated client           |
| Observability        | Makes feature measurable/supportable              | Traces, dashboards, usage events               |
| Performance/scaling  | Removes limit blocking bigger usage               | Async jobs, pagination, cache, streaming       |

# 14. Core IP upgrade patterns

When the goal is “find core IP and upgrade it,” the agent should look for one of these upgrade vectors:

## 14.1 Make the core engine measurable

```text
Before:
  The engine produces outputs.

After:
  The engine produces outputs plus scores, traces, comparisons, regressions,
  and benchmark reports.
```

Feature examples:

```text
evaluation harness
golden test corpus
quality dashboard
run comparison
regression detection
confidence scoring
```

## 14.2 Make the core engine explainable

```text
Before:
  The system gives an answer.

After:
  The system explains why, with inputs, steps, rules, tradeoffs, and evidence.
```

Feature examples:

```text
decision trace
explainability panel
provenance graph
source attribution
intermediate state viewer
```

## 14.3 Make the core engine reusable

```text
Before:
  The core capability is trapped in one UI or CLI.

After:
  It is available through API, SDK, plugins, templates, or workflows.
```

Feature examples:

```text
public API
OpenAPI spec
generated TypeScript SDK
Rust crate boundary cleanup
plugin interface
template registry
```

## 14.4 Make the core engine learn from usage

```text
Before:
  Runs are isolated.

After:
  The system accumulates feedback, examples, preferences, and outcomes.
```

Feature examples:

```text
feedback capture
labeled examples
human review queue
quality trends
recommendation memory
```

## 14.5 Make the core engine collaborative

```text
Before:
  One user runs or configures something.

After:
  Teams can review, comment, approve, compare, and audit.
```

Feature examples:

```text
shared projects
comments
review states
approval workflow
audit trail
role-based access
```

# 15. What I would actually build first

For a real implementation, I would not start with a huge graph database system. I would build in stages.

## Stage 1: Repository cognition MVP

```text
- Parse canonical files.
- Build file/module/API/table/test inventory.
- Add BM25 search.
- Add embeddings for docs/code comments/tests.
- Add basic graph in SQLite/Postgres.
- Generate mission summary and capability map.
```

Deliverable:

```text
repo_context_report.md
capability_graph.json
feature_candidates.md
```

## Stage 2: Precise code/workflow map

```text
- Add Tree-sitter parsing.
- Add LSP or SCIP symbol references.
- Add route → component → API client mapping.
- Add backend endpoint → handler → repository → SQL mapping.
- Add migration/schema extraction.
```

Deliverable:

```text
workflow_map.md
api_data_lineage_graph.json
```

## Stage 3: Runtime validation

```text
- Boot Docker Compose.
- Run app smoke tests.
- Use Playwright to crawl routes.
- Capture screenshots, network calls, console errors.
- Attach traces to workflow graph.
```

Deliverable:

```text
runtime_workflow_report.md
screenshots/
playwright-traces/
```

## Stage 4: Feature scoring and RFC generation

```text
- Generate 10-30 candidate features.
- Score them.
```

Deliverable:

# 16. The strongest retrieval pattern

For a feature candidate, retrieval should always gather this set:

```text
1. Mission evidence
   README, docs, roadmap, examples

2. Current capability evidence
   routes, APIs, components, commands, tests

3. Data evidence
   tables, migrations, schemas, DTOs

4. Workflow evidence
   UI routes, Playwright traces, API calls

5. Core IP evidence
   central modules, engine crates, examples, benchmarks

6. Constraint evidence
   Docker, CI, env, deployment, auth, permissions

7. Proof evidence
   existing tests and missing tests
```

In pseudo-query form:

```text
candidate: "Add run comparison"

retrieve:
  docs where concepts similar to "compare, evaluate, run, result, metric"
  API endpoints matching "run/result/metric"
  React routes/components matching "Run"
  SQL tables matching "runs/results/metrics"
  Rust symbols matching "RunResult, Metric, Evaluation"
  tests matching "run/result/metric"
  Docker/CI commands needed to test
```

Then traverse graph paths:

```text
Mission concept:evaluation
  -> Capability:run execution
  -> Entity:RunResult
  -> Table:runs
  -> Endpoint:GET /runs/:id
  -> Page:RunDetail
  -> Missing:CompareRunPage
```

That is the kind of context-smart reasoning you want.

# 17. Anti-patterns to avoid

```text
Naive vector RAG only:
  Will retrieve semantically related text but miss exact code relationships.

One giant summary:
  Will blur important implementation constraints.

Only reading README:
  Produces generic product ideas disconnected from code.

Only reading code:
  Produces technically feasible ideas disconnected from mission.

Only looking at recent files:
  Misses core IP and long-lived product direction.

Only generating ideas:
  Creates strategy theater without implementation proof.

Only implementing:
  Creates random features without product justification.

No critic:
  The agent will overrate flashy, under-evidenced features.

No scoring history:
  The system cannot learn which ideas were good or bad.
```

# 18. The final mental model

The best system is not “RAG over a repo.” It is a **repo intelligence platform** with agents on top.

```text
RAG answers:
  “Where is something like this mentioned?”

Graph answers:
  “How is this connected to the rest of the system?”

LSP/SCIP answers:
  “What code precisely defines and uses this?”

Runtime traces answer:
  “What actually happens when the product runs?”

Tests answer:
  “What behavior is currently proven?”

Mission synthesis answers:
  “Why should this matter?”

Scoring answers:
  “Is this worth doing now?”

Planning answers:
  “How do we ship it safely?”
```

My strongest recommendation: build the feature agent around **capability graphs**, not files. Files are evidence. Capabilities are the unit of strategy. The agent’s job is to discover where the project’s mission, core IP, workflows, APIs, data model, and tests all point toward the same missing capability — then produce a vertical slice that proves it.
