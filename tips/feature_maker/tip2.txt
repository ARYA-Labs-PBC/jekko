Yes — the ambitious version is a **capability-upgrade agent**, not a bug-finder. Your bug-finder baseline is invariant/counterexample-driven: infer what the repo promises, find where the promise breaks, prove it with a failing case, then patch minimally.  The feature-upgrade version should do the analogous higher-level thing:

> “What capability does this project’s mission, architecture, domain model, data, and user workflow imply — but not yet deliver — and can I design a credible, testable path to add it?”

That means the core product is not “RAG over a repo.” It is a **repo intelligence system**: graph + hybrid search + code intelligence + runtime/test evidence + planning agents + verification harness.

## 1. Canonical files to inspect in a Rust/Vite/TypeScript/React/SQL/Docker repo

The agent should start from a canonical file taxonomy, because each file type tells a different story: mission, domain, build surface, runtime surface, user surface, API surface, data surface, and proof surface.

| Layer                                  | Canonical files / directories                                                                                                                                                                                                                                                                 | What they reveal                                                                                                                                                                                                                                                        |
| -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Mission / product intent**           | `README.md`, `docs/`, `MISSION.md`, `VISION.md`, `MOONSHOT.md`, `ROADMAP.md`, `CHANGELOG.md`, `CONTRIBUTING.md`, `SECURITY.md`, `LICENSE`, `CODEOWNERS`, `TODO.md`, `NOTES.md`, `docs/rfcs/`, `docs/adr/`                                                                                     | What the project claims to be, who it serves, what is unfinished, what risks or constraints matter. ADRs are especially useful because they capture significant architectural decisions, context, and consequences. ([Architectural Decision Records][1])               |
| **Rust backend**                       | `Cargo.toml`, `Cargo.lock`, `rust-toolchain.toml`, `.cargo/config.toml`, `src/main.rs`, `src/lib.rs`, `src/bin/`, `crates/*/Cargo.toml`, `build.rs`, `examples/`, `benches/`, `tests/`, `sqlx-data.json`                                                                                      | Crate boundaries, binaries, libraries, features, dependencies, internal APIs, public APIs, feature flags, and test intent. `Cargo.toml` is the Cargo manifest containing package metadata needed to compile a Rust package. ([Rust Documentation][2])                   |
| **TypeScript / React / Vite frontend** | `package.json`, lockfile, `tsconfig*.json`, `vite.config.ts`, `index.html`, `src/main.tsx`, `src/App.tsx`, `src/routes/`, `src/pages/`, `src/components/`, `src/hooks/`, `src/lib/`, `src/api/`, `src/stores/`, `vitest.config.ts`, `playwright.config.ts`, `.eslintrc*`, `tailwind.config.*` | App entrypoint, routes, screens, user flows, UI state, API clients, generated types, build scripts, and quality gates. `package.json` carries package metadata, dependencies, and scripts; Vite explicitly supports `vite.config.ts`. ([npm Docs][3])                   |
| **API / contracts**                    | `openapi.yaml`, `openapi.json`, `asyncapi.yaml`, `graphql.schema`, `schema.graphql`, `proto/`, `buf.yaml`, `api/`, `contracts/`, generated clients, SDK folders                                                                                                                               | What the project exposes, consumes, promises, and could platformize. OpenAPI gives a language-agnostic description of HTTP APIs, while AsyncAPI describes event-driven APIs. ([Swagger][4])                                                                             |
| **SQL / data model**                   | `migrations/`, `schema.sql`, `seed.sql`, `fixtures/`, `db/`, `database/`, `prisma/schema.prisma`, `diesel.toml`, `sqlx.toml`, query files, RLS policy files                                                                                                                                   | Durable domain model, invariants, capability gaps, unexposed workflows, missing admin screens, missing analytics, possible IP. PostgreSQL’s information schema and system catalogs are useful for introspecting database objects and schema metadata. ([PostgreSQL][5]) |
| **Docker / runtime / deployment**      | `Dockerfile`, `.dockerignore`, `docker-compose.yml`, `compose.yaml`, `compose.override.yml`, `entrypoint.sh`, `scripts/`, `.env.example`, `.env.template`, `k8s/`, `helm/`, `terraform/`, `fly.toml`, `render.yaml`, `railway.json`                                                           | How the project actually runs, what services exist, whether dev/prod differ, what background jobs and databases exist. A Dockerfile is a text document of commands Docker reads to assemble an image. ([Docker Documentation][6])                                       |
| **CI / proof surface**                 | `.github/workflows/*.yml`, `.gitlab-ci.yml`, `Makefile`, `justfile`, `Taskfile.yml`, `scripts/test*`, `scripts/build*`, `coverage/`, `clippy.toml`                                                                                                                                            | How maintainers prove correctness. GitHub Actions workflows live in `.github/workflows`, are YAML files checked into the repo, and can build, test, deploy, or run scheduled/manual tasks. ([GitHub Docs][7])                                                           |
| **Observability / operations**         | `otel-config.yaml`, `prometheus.yml`, `grafana/`, `dashboards/`, `runbooks/`, `alerts/`, `SLO.md`, `ops/`                                                                                                                                                                                     | Production reality, success metrics, bottlenecks, and safe rollout strategy. OpenTelemetry provides APIs, SDKs, agents, and collectors for traces, metrics, and logs. ([OpenTelemetry][8])                                                                              |
| **Architecture maps**                  | `docs/architecture.md`, `docs/diagrams/`, `c4/`, `structurizr.dsl`, `adr/`, `rfcs/`                                                                                                                                                                                                           | System boundaries, containers, components, dependencies, and historical design intent. The C4 model is explicitly organized around context, containers, components, and code. ([C4 model][9])                                                                           |

The agent should not treat these files equally. `README.md`, `docs/`, root manifests, API contracts, database migrations, app entrypoints, route definitions, and CI workflows are **orientation files**. They establish the project’s shape before the agent goes deep.

## 3. Best architecture for the agent system

Build this as a layered intelligence system, not as one giant LLM prompt.

```text
                 ┌──────────────────────────────┐
                 │   Feature Opportunity Agent   │
                 └──────────────┬───────────────┘
                                │
          ┌─────────────────────┼─────────────────────┐
          │                     │                     │
  Mission/Docs Agent     Architecture Agent     User Workflow Agent
          │                     │                     │
          └─────────────┬───────┴───────────┬─────────┘
                        │                   │
                 Capability Graph      Hybrid Retrieval
                        │                   │
         ┌──────────────┴──────────────┐    │
         │                             │    │
   Code/Symbol Graph              Runtime/Test Evidence
         │                             │
  AST/LSP/CodeQL/CPG              CI/logs/OTel/tests
```

The important design choice: **all agent conclusions must be grounded in artifacts**. Every proposed feature should point to docs, code, schema, tests, user flows, or runtime evidence.

### Layer A — repository cartographer

This layer classifies the repo before reasoning.

It should detect:

```text
workspace layout
package manager
Rust crates
frontend apps
backend apps
database engine
migration tool
API contract files
Docker services
CI commands
test commands
route structure
auth/session model
external integrations
observability setup
```

For Rust, use `cargo metadata`, `Cargo.toml`, crate targets, features, and source entrypoints. For TypeScript/React/Vite, use `package.json`, lockfiles, `tsconfig`, `vite.config.ts`, route files, and generated clients. For Docker/CI, inspect Dockerfiles, Compose services, GitHub Actions workflows, and shell scripts.

### Layer B — code intelligence

Use multiple code-understanding technologies because no single index is enough.

Recommended tools:

| Need                                 | Technology                                                                                                                                                                                                  |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Syntax trees and structural chunking | **Tree-sitter**. It is a parser generator and incremental parsing library that can build concrete syntax trees and update them efficiently. ([Tree-sitter][10])                                             |
| Symbols, references, definitions     | **Language Server Protocol**, `rust-analyzer`, TypeScript language service. LSP supports IDE-like features such as autocomplete, go-to-definition, and find-all-references. ([Microsoft GitHub][11])        |
| Semantic/static code queries         | **CodeQL**, **Joern**, **Semgrep**. CodeQL lets you query code as data; Joern builds code property graphs for cross-language analysis; Semgrep is useful for pattern and guardrail scanning. ([CodeQL][12]) |
| API contracts                        | OpenAPI, AsyncAPI, GraphQL schema, protobuf parsers.                                                                                                                                                        |
| SQL schema                           | Migration parser, live introspection through `information_schema`, generated schema graph.                                                                                                                  |
| UI flows                             | React route extraction, Playwright crawling, Storybook indexing if present.                                                                                                                                 |
| Dependency/build graph               | `cargo metadata`, npm/pnpm/yarn lockfile parser, Docker Compose parser, CI workflow parser.                                                                                                                 |

The best system stores **symbol-level facts**, not only chunks. For example:

```text
Function: create_invoice
Defined in: crates/billing/src/invoice.rs
Calls: insert_invoice, authorize_workspace, emit_event
Writes: invoices, invoice_events
Tested by: billing_creates_invoice
Rendered by: /billing/new route
Observed by: billing.invoice.created metric
```

That is far more useful than a vector chunk saying “this file is about invoices.”

### Layer C — graph database

Use a graph database for relationships and traversal, not for text search. Neo4j is a natural fit because property graphs model nodes, relationships, and properties directly. ([Graph Database & Analytics][13])

A graph DB helps answer questions like:

```cypher
MATCH (c:Capability)-[:IMPLEMENTED_BY]->(e:Endpoint)
WHERE NOT (c)-[:HAS_SCREEN]->(:Screen)
RETURN c, e
```

or:

```cypher
MATCH (d:DocClaim)-[:PROMISES]->(c:Capability)
WHERE NOT (c)-[:TESTED_BY]->(:Test)
RETURN d, c
```

For small/local setups, a graph can start as SQLite/Postgres tables. For serious multi-repo intelligence, use Neo4j, Memgraph, FalkorDB, ArangoDB, or a custom property graph service.

### Layer D — hybrid retrieval, not plain vector search

Use hybrid search because feature discovery needs both exact terms and semantic concepts. Exact terms matter for things like `tenant_id`, `billing`, `MOONSHOT`, `invoice`, `dangerouslySetInnerHTML`, `VITE_`, `openapi.yaml`, or `workspace_id`. Semantic retrieval matters for concepts like “onboarding,” “manual approval,” “workflow automation,” or “export capability.”

Recommended retrieval stack:

```text
BM25 / lexical index
+ vector embeddings
+ metadata filters
+ graph-neighborhood expansion
+ cross-encoder or LLM reranking
+ provenance/citation tracking
```

Vespa is attractive for large-scale hybrid retrieval because its docs explicitly cover combining BM25 with vector search; Qdrant is a strong vector-search component and supports vector similarity search with filtering. ([Vespa. Big data. Real time. Open source.][14])

For an MVP:

```text
Postgres + pgvector
Tantivy or ripgrep/BM25
Tree-sitter chunks
cargo metadata
ts-morph / TypeScript compiler API
Neo4j optional
```

For production:

```text
Neo4j or equivalent graph DB
Vespa / Elasticsearch / OpenSearch for hybrid search
Qdrant or Milvus if using a dedicated vector DB
Object store for raw artifacts
Temporal or LangGraph for long-running workflows
MCP servers for repo, DB, CI, browser, GitHub, docs
```

### Layer E — runtime and user-flow evidence

A feature agent should not only read code. It should run the product.

Use:

```text
Docker Compose startup
Playwright route crawler
API smoke tests
OpenAPI request generation
database seed inspection
CI workflow replay
logs and traces
screenshots of major flows
```

OpenTelemetry is valuable because traces, metrics, and logs expose what actually runs and where users or workflows spend time. ([OpenTelemetry][8])

### Layer F — orchestration

You need durable, resumable, reviewable workflows. LangGraph is designed around agent orchestration features such as durable execution, streaming, and human-in-the-loop workflows; Temporal is a durable execution platform for reliable workflows and retries. ([LangChain Docs][15])

MCP is useful as a tool boundary: it standardizes how LLM clients access external context such as files, schemas, and application-specific resources. ([Model Context Protocol][16])

My practical recommendation:

```text
MVP orchestration:
  LangGraph or a simple state-machine runner

Serious production orchestration:
  Temporal for durable workflow execution
  LangGraph-style state graph for agent logic
  MCP servers for repo, browser, DB, CI, GitHub, graph, search
```

## 5. Feature opportunity categories

I would classify proposed upgrades into these categories:

| Category                               | What it looks like                                                                      | Example                                                                              |
| -------------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **Latent capability completion**       | Repo already has partial backend/data support but lacks full product workflow           | “There are import tables and parser code, but no UI for bulk import.”                |
| **Core IP productization**             | Internal engine exists but is not exposed as a reusable capability                      | “Turn the internal rules/scoring engine into an API, SDK, or plugin system.”         |
| **Workflow automation**                | Users repeat a multi-step manual flow                                                   | “Add saved workflows, approval automation, or batch execution.”                      |
| **API/platform expansion**             | App is usable only through UI, but domain model suggests external integrations          | “Expose public API, webhooks, SDK, API keys, usage dashboard.”                       |
| **Data/analytics upgrade**             | Tables/events exist but no insight layer                                                | “Add operational analytics, trend reports, anomaly detection, cohort views.”         |
| **Collaboration/enterprise readiness** | Product has user/project concepts but weak org workflows                                | “Roles, audit log, comments, assignments, review queues, SSO-ready model.”           |
| **Integration expansion**              | Existing integration patterns can be generalized                                        | “Plugin marketplace, connector framework, import/export adapters.”                   |
| **Reliability/scale upgrade**          | Architecture implies future scale but lacks queues, retries, idempotency, observability | “Move long jobs to queue with progress UI and resumability.”                         |
| **Developer experience upgrade**       | Project has hard setup, weak docs, poor local tooling                                   | “One-command dev environment, generated API clients, contract tests.”                |
| **Security/compliance upgrade**        | Product handles sensitive data but lacks governance                                     | “Audit trails, permission matrix, data retention, export/delete workflows.”          |
| **AI/product intelligence upgrade**    | Domain model has enough structure for useful AI assist                                  | “AI assistant grounded in project data, but only after permissions and evals exist.” |

The best opportunities are usually not “add AI.” They are:

```text
complete a latent capability
productize internal IP
automate a painful workflow
expose a platform surface
turn existing data into decisions
```

## 6. How to score feature opportunities

Use a weighted score plus veto rules.

```text
Net Score =
  Strategic Fit
+ User Value
+ Core-IP Leverage
+ Repo Affordance
+ Evidence Strength
+ Technical Feasibility
+ Differentiation
+ Verification Clarity
+ Rollout Safety
- Risk Penalty
```

Suggested weights:

| Dimension                 | Weight | What to measure                                                 |
| ------------------------- | -----: | --------------------------------------------------------------- |
| **Strategic fit**         |     15 | Does README/docs/mission/roadmap imply this?                    |
| **User value**            |     15 | Does it remove pain, unlock a workflow, or expand usage?        |
| **Core-IP leverage**      |     15 | Does it amplify the project’s unique engine/data/model?         |
| **Repo affordance**       |     12 | Is the repo already structurally close to supporting it?        |
| **Evidence strength**     |     10 | Is the proposal grounded in docs/code/schema/tests/issues?      |
| **Technical feasibility** |     10 | Can it be built incrementally without a rewrite?                |
| **Differentiation/moat**  |      8 | Does it create something hard to copy?                          |
| **Verification clarity**  |      8 | Can tests/evals/metrics prove it works?                         |
| **Rollout safety**        |      7 | Can it be feature-flagged, migrated, observed, and rolled back? |

Risk penalty:

```text
security/privacy risk
migration risk
scope explosion
dependency risk
architecture mismatch
unclear ownership
poor testability
high operational burden
```

Grade rubric:

```text
A: Strong mission fit, high repo affordance, clear core-IP leverage, testable vertical slice.
B: Valuable and feasible, but missing one evidence category or has moderate risk.
C: Interesting but speculative; needs discovery spike.
D: Generic, weakly grounded, too broad, or not verifiable.
Veto: No evidence, no verification path, requires rewrite, violates security/privacy constraints.
```

## 7. How the agent should find “core IP”

Core IP is rarely labeled “core IP.” The agent should infer it.

Signals:

```text
high fan-in domain modules
custom algorithms
domain-specific terminology repeated across docs/code/schema
handwritten logic instead of framework CRUD
central tables with many workflows
custom scoring/ranking/planning/simulation
complex state machines
integration logic with many edge cases
unique data transformations
benchmarks/evaluation harnesses
operator-only internal tools
```

Graph query examples:

```text
Find modules with high centrality that are not generic infrastructure.
Find domain terms appearing in README + SQL + Rust + UI.
Find internal-only capabilities with no public API or UI.
Find algorithms with tests but no product surface.
Find tables that drive many workflows but have no analytics/reporting screen.
```

Core IP upgrade patterns:

| IP type                   | Upgrade pattern                                                          |
| ------------------------- | ------------------------------------------------------------------------ |
| Rules/scoring engine      | Add explainability, API, SDK, versioned rules, eval suite                |
| Workflow engine           | Add visual workflow builder, automation templates, audit logs            |
| Domain ontology           | Add schema browser, semantic search, validation rules, import/export     |
| Integration layer         | Add connector SDK, plugin framework, marketplace, webhook system         |
| Planning/simulation logic | Add scenario planner, comparisons, batch runs, saved experiments         |
| Data pipeline             | Add quality checks, lineage, replay, observability, user-facing insights |
| Internal admin tool       | Productize as enterprise control plane                                   |
| Evaluation harness        | Turn into benchmark suite and quality gate                               |

A strong “core IP upgrade” proposal should include:

```yaml
ip_asset: "internal scoring/rules engine"
current_surface: "used by backend only"
missing_surface: "no API, no UI explanation, no eval dashboard"
upgrade: "versioned scoring engine with explanations + eval harness"
why_now: "schema and tests already contain scoring concepts"
first_slice: "explain one score in UI and verify against golden fixtures"
moat: "domain-specific explainable decisions and benchmark history"
```

## 9. The best feature-discovery loops

### Loop A — Mission-to-code gap

```text
1. Extract promised capabilities from README/docs/roadmap.
2. Map each promise to code, API, UI, DB, tests, and telemetry.
3. Find promises with partial implementation.
4. Rank by user value and implementation closeness.
```

Example finding:

```text
README says “collaborative workspace.”
DB has workspaces and memberships.
Backend has membership APIs.
Frontend has no invite flow, role editor, or audit trail.
Opportunity: enterprise collaboration upgrade.
```

### Loop B — Data-model-to-product gap

```text
1. Read schema and migrations.
2. Identify central domain entities.
3. Check which entities have routes, APIs, docs, and tests.
4. Find underexposed entities.
```

Example:

```text
DB has job_runs, job_steps, failure_reason, retry_count.
No UI shows job progress or failures.
Opportunity: job observability dashboard.
```

### Loop C — Workflow friction miner

```text
1. Crawl UI routes and forms.
2. Identify repeated manual sequences.
3. Check whether batch, saved, scheduled, or templated variants exist.
4. Propose automation.
```

Example:

```text
Users create the same report filters repeatedly.
Opportunity: saved report templates + scheduled delivery.
```

### Loop D — API/platformization miner

```text
1. Compare backend handlers with OpenAPI/public docs.
2. Find internal capabilities not exposed externally.
3. Check auth, rate-limit, audit, and API-key readiness.
4. Propose platform surface.
```

Example:

```text
Internal import/export exists.
No public API, no webhook, no SDK.
Opportunity: developer platform upgrade.
```

### Loop E — Core-IP upgrade miner

```text
1. Find central custom domain engine.
2. Identify current private/internal use.
3. Find missing explainability, configuration, versioning, evals, or public surface.
4. Propose a feature that compounds the unique asset.
```

Example:

```text
Custom planner exists.
No scenario comparison, no benchmark suite, no user-facing explanation.
Opportunity: explainable planner + scenario lab.
```

## 11. Implementation planning: from idea to verified work plan

The agent should produce a plan in **thin vertical slices**, not a big rewrite.

Example structure:

```text
Slice 1: schema + contract + failing/expected tests
Slice 2: backend endpoint/service implementation
Slice 3: frontend route/component workflow
Slice 4: e2e test and seed fixture
Slice 5: observability, docs, and rollout
```

Verification matrix:

| Area             | Proof                                                  |
| ---------------- | ------------------------------------------------------ |
| Backend behavior | Rust unit/integration tests                            |
| API contract     | OpenAPI/GraphQL/protobuf contract tests                |
| SQL correctness  | Migration apply/rollback, constraint tests, seed tests |
| UI workflow      | Playwright e2e test                                    |
| Permissions      | negative authz tests                                   |
| Runtime          | Docker Compose smoke test                              |
| Performance      | targeted benchmark/load check                          |
| Observability    | trace/span/metric/log assertion                        |
| Rollout          | feature flag and rollback path                         |
| Docs             | README/API docs updated                                |

For larger core-IP upgrades, add an eval suite:

```text
golden fixtures
expected decisions
regression benchmark
explanation snapshots
performance budget
edge-case matrix
```

## 13. What makes the system consistently good

The system needs hard rules.

```text
Never propose a feature without evidence from at least two artifact classes.
Never propose a broad rewrite as the first slice.
Never propose generic “AI assistant” unless the repo has data, permissions, evals, and a grounded workflow.
Never ignore DB/migration impact.
Never ignore rollout and observability.
Never ignore existing roadmap/non-goals.
Never treat frontend-only capability as implemented product capability.
Never treat backend-only capability as user-visible capability.
Never treat TypeScript types as proof of runtime contract.
Never treat docs as proof of implementation.
```

Minimum evidence bar:

```text
A proposal must cite:
1. product intent or user need
2. current architecture affordance
3. code/schema/API evidence
4. missing workflow/surface
5. verification path
```

Best feature proposals usually sound like:

```text
“The project already has 70% of this capability structurally, but users cannot benefit from it yet.”
```

Weak proposals sound like:

```text
“Add AI.”
“Add dashboard.”
“Improve UX.”
“Make it scalable.”
```

## 15. The deepest heuristic

For bugs, your canonical question was:

```text
What invariant does this code promise, and where is it not enforced?
```

For major feature upgrades, the canonical question should be:

```text
What capability does this project’s structure imply, and where is that capability not yet productized?
```

That is the high-value move. The best agent will find features that feel obvious in hindsight because the repo was already pointing at them: the docs hinted at them, the data model prepared for them, the backend partially implemented them, the frontend lacked them, and the tests did not yet encode them.
