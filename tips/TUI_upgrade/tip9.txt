# Engineering Report: Brand-New TUI Direction With Ratatui Power

## Executive recommendation

The uploaded TUI codebase is **not currently Rust/Ratatui**. It is a **TypeScript + SolidJS + OpenTUI** frontend using `@opentui/core`, `@opentui/solid`, Solid stores, backend SDK/SSE sync, a plugin slot runtime, and Hono backend routes. That matters because “leveraging Ratatui” is not a small package swap. It is either:

1. **Fast path:** redesign the existing OpenTUI/Solid UI now using the file-level changes below.
2. **Best path for the world’s greatest Rust TUI:** build a new native Rust/Ratatui frontend that consumes the existing backend API and event stream.

I strongly recommend the second path as the strategic direction. Keep the backend, server routes, SDK semantics, session/event model, permissions, MCP, LSP, daemon, provider, and file APIs. Replace the visual frontend with a new Rust app powered by Ratatui. Ratatui is a Rust TUI library with widgets and utilities for complex terminal UIs, and it uses `crossterm` as the default backend in its quickstart/docs. ([Docs.rs][1])

The design should not be a “better OpenCode.” It should feel like a **native terminal command center for engineering**: left intelligence rail, center coding canvas, bottom command dock, top command bar, beautiful dark/white themes, premium typography guidance, and a recognizable product identity that has no OpenCode visual DNA.

# 3. Ratatui strategy

## Why Ratatui is a strong fit

Ratatui gives you native Rust ownership over layout, styling, rendering, and custom widgets. It has built-in widgets such as `Block`, `Paragraph`, `List`, `Table`, `Tabs`, `Gauge`, `Sparkline`, `Canvas`, `Chart`, `Scrollbar`, and more, which map directly to the proposed command bar, left rail, timeline, command dock, and dashboard panels. ([Docs.rs][2])

The key architectural shift: Ratatui is immediate-mode. Each frame is rendered from application state. It does not provide your input/event system; your app owns keyboard input, mouse input, state updates, and deciding which widgets reflect that state. ([Ratatui][3])

That is actually perfect here because your backend/event model is already separate. The Rust frontend can maintain a clean `AppState`, consume SSE/backend events, and render the whole UI every frame.

## App model

```rust
// src/app.rs

use crate::{
    action::Action,
    backend::model::*,
    state::{PromptState, SessionState, SyncState},
    theme::{Theme, ThemeMode},
};

pub struct App {
    pub running: bool,
    pub theme_mode: ThemeMode,
    pub theme: Theme,

    pub current_session_id: Option<String>,
    pub sessions: Vec<Session>,
    pub session: SessionState,
    pub sync: SyncState,
    pub prompt: PromptState,

    pub left_rail_visible: bool,
    pub left_rail_mode: RailMode,
    pub focus: Focus,
    pub dialogs: DialogStack,
    pub toast: Option<Toast>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Focus {
    Canvas,
    CommandDock,
    LeftRail,
    Dialog,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum RailMode {
    Auto,
    Hidden,
    Forced,
}

impl App {
    pub fn update(&mut self, action: Action) {
        match action {
            Action::Quit => self.running = false,

            Action::ToggleThemeMode => {
                self.theme_mode = self.theme_mode.toggle();
                self.theme = Theme::nova(self.theme_mode);
            }

            Action::ToggleLeftRail => {
                self.left_rail_visible = !self.left_rail_visible;
                self.left_rail_mode = if self.left_rail_visible {
                    RailMode::Forced
                } else {
                    RailMode::Hidden
                };
            }

            Action::PromptChanged(text) => {
                self.prompt.text = text;
            }

            Action::SubmitPrompt => {
                self.prompt.submit_requested = true;
            }

            Action::BackendEvent(evt) => {
                self.sync.apply(evt);
            }

            _ => {}
        }
    }
}
```

## Input mapping

```rust
// src/input/keymap.rs

use crossterm::event::{KeyCode, KeyEvent, KeyModifiers};
use crate::action::Action;

pub fn map_key(key: KeyEvent) -> Option<Action> {
    match (key.modifiers, key.code) {
        (KeyModifiers::CONTROL, KeyCode::Char('c')) => Some(Action::Quit),

        // White/Dark shortcut
        (KeyModifiers::CONTROL, KeyCode::Char('w')) => Some(Action::ToggleThemeMode),

        // Left rail toggle
        (KeyModifiers::CONTROL, KeyCode::Char('b')) => Some(Action::ToggleLeftRail),

        (KeyModifiers::CONTROL, KeyCode::Char('p')) => Some(Action::OpenCommandPalette),
        (KeyModifiers::CONTROL, KeyCode::Char('t')) => Some(Action::OpenThemePicker),

        (_, KeyCode::Esc) => Some(Action::OpenCommandPalette),
        (_, KeyCode::PageUp) => Some(Action::PageCanvasUp),
        (_, KeyCode::PageDown) => Some(Action::PageCanvasDown),

        _ => None,
    }
}
```

For the existing app’s leader-key system, keep:

```text
Ctrl+X W = White/Dark mode
Ctrl+X B = Left rail
Ctrl+X T = Theme picker
```

For the native Ratatui app, I would additionally support direct terminal shortcuts:

```text
Ctrl+W = White/Dark mode
Ctrl+B = Left rail
Ctrl+P = Command palette
```

That gives power users instant access without making the UI feel borrowed.

# 5. Ratatui layout implementation

## Main shell

```rust
// src/ui/shell.rs

use ratatui::{
    Frame,
    layout::{Constraint, Direction, Layout, Rect},
};

use crate::{app::App, ui};

pub fn render(frame: &mut Frame, app: &mut App) {
    let area = frame.area();

    let [top, body, dock] = Layout::vertical([
        Constraint::Length(1),
        Constraint::Min(1),
        Constraint::Length(app.command_dock_height()),
    ])
    .areas(area);

    ui::top_bar::render(frame, app, top);

    let rail_width = app.left_rail_width(area.width);
    let show_rail = app.should_show_left_rail(area.width);

    if show_rail {
        let [rail, canvas] = Layout::horizontal([
            Constraint::Length(rail_width),
            Constraint::Min(1),
        ])
        .areas(body);

        ui::left_rail::render(frame, app, rail);
        ui::canvas::render(frame, app, canvas);
    } else {
        ui::canvas::render(frame, app, body);
    }

    ui::command_dock::render(frame, app, dock);

    if let Some(dialog) = app.dialogs.current() {
        ui::dialogs::render(frame, app, dialog);
    }

    if let Some(toast) = &app.toast {
        ui::status::render_toast(frame, app, toast);
    }
}
```

## Left rail

```rust
// src/ui/left_rail.rs

use ratatui::{
    Frame,
    layout::{Constraint, Layout, Rect},
    style::{Modifier, Style, Stylize},
    text::{Line, Span},
    widgets::{Block, Borders, Gauge, List, ListItem, Paragraph, Sparkline},
};

use crate::app::App;

pub fn render(frame: &mut Frame, app: &App, area: Rect) {
    let theme = &app.theme;

    let block = Block::default()
        .borders(Borders::RIGHT)
        .border_style(Style::default().fg(theme.border_subtle))
        .style(Style::default().bg(theme.panel));

    frame.render_widget(block, area);

    let inner = area.inner(ratatui::layout::Margin {
        horizontal: 1,
        vertical: 1,
    });

    let [header, session, files, tools, health] = Layout::vertical([
        Constraint::Length(3),
        Constraint::Length(5),
        Constraint::Min(6),
        Constraint::Length(8),
        Constraint::Length(4),
    ])
    .areas(inner);

    frame.render_widget(
        Paragraph::new(vec![
            Line::from(vec![
                Span::styled("◆", Style::default().fg(theme.primary)),
                Span::raw(" WORKSPACE"),
            ])
            .bold(),
            Line::from(app.workspace_label()).fg(theme.text_muted),
        ])
        .style(Style::default().bg(theme.panel)),
        header,
    );

    frame.render_widget(
        Paragraph::new(app.session_summary())
            .block(Block::default().title("Session").borders(Borders::TOP))
            .style(Style::default().fg(theme.text).bg(theme.panel)),
        session,
    );

    let changed_files: Vec<ListItem> = app
        .session
        .changed_files
        .iter()
        .map(|f| ListItem::new(Line::from(vec![
            Span::styled("▸ ", Style::default().fg(theme.info)),
            Span::raw(&f.path),
            Span::styled(format!(" +{} -{}", f.additions, f.deletions), Style::default().fg(theme.text_muted)),
        ])))
        .collect();

    frame.render_widget(
        List::new(changed_files)
            .block(Block::default().title("Changed files").borders(Borders::TOP))
            .style(Style::default().fg(theme.text).bg(theme.panel)),
        files,
    );

    frame.render_widget(
        Sparkline::default()
            .data(&app.session.activity_sparkline)
            .style(Style::default().fg(theme.secondary).bg(theme.panel)),
        tools,
    );

    frame.render_widget(
        Gauge::default()
            .ratio(app.session.context_ratio())
            .label(format!("context {:.0}%", app.session.context_ratio() * 100.0))
            .gauge_style(Style::default().fg(theme.primary).bg(theme.elevated)),
        health,
    );
}
```

This is where Ratatui shines: `List`, `Sparkline`, `Gauge`, `Block`, and `Paragraph` give the left rail a polished dashboard feel without custom terminal rendering hacks. ([Docs.rs][2])

## 7.2 Replace the right sidebar with a left intelligence rail

```tsx
export function MessageCard(props: {
  tone?: "user" | "assistant" | "tool" | "error" | "thinking"
  title?: JSX.Element | string
  children: JSX.Element
  marginTop?: number
  onMouseUp?: () => void
}) {
  const { theme } = useTheme()

  const color = () => {
    if (props.tone === "user") return theme.primary
    if (props.tone === "assistant") return theme.secondary
    if (props.tone === "tool") return theme.info
    if (props.tone === "error") return theme.error
    if (props.tone === "thinking") return theme.textMuted
    return theme.border
  }

  return (
    <box marginTop={props.marginTop ?? 1} gap={0} onMouseUp={props.onMouseUp}>
      <Show when={props.title}>
        <box flexDirection="row" gap={1} paddingLeft={1}>
          <text fg={color()}>
            <b>◆</b>
          </text>
          <text fg={theme.textMuted}>{props.title}</text>
        </box>
      </Show>

      <box
        backgroundColor={theme.backgroundPanel}
        border={["top"]}
        borderColor={color()}
        paddingLeft={2}
        paddingRight={2}
        paddingTop={1}
        paddingBottom={1}
      >
        {props.children}
      </box>
    </box>
  )
}
```

This breaks the OpenCode-like visual pattern and creates a richer engineering timeline.

1. **Fix backend `/open-themes` bug** and command fallback.

5. **Move session sidebar to the left.**
