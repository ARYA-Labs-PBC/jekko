Yes — **if the goal is the world’s greatest Rust TUI app, I would make Ratatui the strategic frontend target.** The uploaded fork is currently a TypeScript/SolidJS/OpenTUI frontend, so Ratatui cannot be “dropped into” the existing `.tsx` files. But the backend/API shape in `TUI_BACKEND_API.md` is already good enough to support a Rust-native Ratatui frontend that consumes the same HTTP + SSE session/event surface.

My revised recommendation is:

1. **Short term:** apply the visual/design changes I outlined to the current OpenTUI/Solid code so you can ship fast.
2. **Strategic path:** build a new Rust `ratatui` frontend as a first-class app, preserving the backend APIs and event stream.
3. **Long term:** retire the recognizable OpenCode/OpenTUI visual structure entirely and make the Ratatui client the flagship.

Ratatui is especially attractive here because it is a Rust crate for complex terminal UIs with widgets, layout, styling, backends, and text primitives; it was forked from `tui-rs` in 2023 and is actively evolving. The current latest line is `0.30.0`, which introduced modularization, backend flexibility, widget/layout improvements, and a newer `ratatui::run()` API. ([Docs.rs][1])

# Ratatui Addendum to the Engineering Report

## 1. New strategic architecture

The current app should become:

```txt
existing backend / SDK / session engine
        │
        ├── existing TypeScript/OpenTUI client, temporary
        │
        └── new Rust Ratatui client, flagship
```

The Ratatui client should consume:

```txt
GET  /event
POST /session/:sessionID/prompt
POST /session/:sessionID/abort
POST /tui/append-prompt
POST /tui/submit-prompt
POST /tui/open-sessions
POST /tui/open-models
POST /tui/open-themes
POST /tui/select-session
POST /tui/publish
```

Do **not** try to recreate the backend inside Rust first. Keep the backend/session engine, build a beautiful Rust TUI shell around it, and progressively replace the current TS TUI.

Ratatui is immediate-mode: the UI is redrawn from application state on each frame. That is a great fit for this app because session streaming, tool progress, prompt state, mode changes, and left rail state can all be modeled as one `AppState` rendered every frame. Ratatui itself does not own your input/event system, so you must design the event loop deliberately with keyboard, mouse, timers, backend SSE, and command events flowing into one app reducer. ([Ratatui][2])

## 6. Event loop

The Ratatui app should merge terminal events, backend events, ticks, render requests, and commands into one channel.

```rust
// src/event.rs

use crossterm::event::KeyEvent;

#[derive(Debug)]
pub enum AppEvent {
    Tick,
    Render,
    Key(KeyEvent),
    Mouse(crossterm::event::MouseEvent),
    Resize { width: u16, height: u16 },
    Backend(BackendEvent),
    Command(Command),
    Error(String),
    Quit,
}
```

Recommended loop:

```rust
// src/main.rs

use color_eyre::Result;
use ratatui::DefaultTerminal;

mod app;
mod event;
mod action;
mod client;
mod layout;
mod theme;
mod widgets;
mod command;

#[tokio::main]
async fn main() -> Result<()> {
    color_eyre::install()?;

    let mut terminal = ratatui::init();
    let result = run(&mut terminal).await;
    ratatui::restore();

    result
}

async fn run(terminal: &mut DefaultTerminal) -> Result<()> {
    let mut app = app::App::new().await?;
    app.start_event_tasks().await?;

    while app.state.running {
        terminal.draw(|frame| {
            widgets::render_app(frame, &app.state);
        })?;

        let event = app.next_event().await?;
        app.handle_event(event).await?;
    }

    Ok(())
}
```

This keeps the frame rendering simple and lets the app redraw after keypresses, resize, backend SSE deltas, prompt edits, model changes, and theme toggles.

## 7. Layout engine

Create one layout function that produces named regions.

```rust
// src/layout/regions.rs

use ratatui::layout::{Constraint, Direction, Layout, Rect};

#[derive(Debug, Clone, Copy)]
pub struct AppRegions {
    pub ribbon: Rect,
    pub body: Rect,
    pub left_rail: Option<Rect>,
    pub transcript: Rect,
    pub composer: Rect,
    pub status: Rect,
}

pub fn compute(area: Rect, left_rail_visible: bool, rail_width: u16) -> AppRegions {
    let vertical = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(1), // command ribbon
            Constraint::Min(6),    // body
            Constraint::Length(5), // composer
            Constraint::Length(1), // status strip
        ])
        .split(area);

    let ribbon = vertical[0];
    let body = vertical[1];
    let composer = vertical[2];
    let status = vertical[3];

    let (left_rail, transcript) = if left_rail_visible && body.width >= 100 {
        let horizontal = Layout::default()
            .direction(Direction::Horizontal)
            .constraints([
                Constraint::Length(rail_width),
                Constraint::Min(40),
            ])
            .split(body);

        (Some(horizontal[0]), horizontal[1])
    } else {
        (None, body)
    };

    AppRegions {
        ribbon,
        body,
        left_rail,
        transcript,
        composer,
        status,
    }
}
```

This is where you guarantee the app is **not** recognized as OpenCode: the left rail is structural, not incidental.

pub fn map_key(key: KeyEvent) -> Option<Command> {
    match (key.code, key.modifiers) {

pub fn apply_command(state: &mut AppState, command: Command) {
    match command {

                ThemeMode::Void => ThemeMode::Snow,
                ThemeMode::Snow => ThemeMode::Void,

            state.toast = Some(ToastState::info(match state.theme_mode {

    }

## 10. Rendering root

```rust
// src/widgets/mod.rs

use ratatui::Frame;
use ratatui::widgets::{Block, Clear};
use crate::app::AppState;
use crate::layout;
use crate::theme::Theme;

pub mod command_ribbon;
pub mod left_rail;
pub mod transcript;
pub mod composer;
pub mod status_strip;
pub mod dialog;
pub mod toast;

pub fn render_app(frame: &mut Frame, state: &AppState) {
    let area = frame.area();
    let theme = Theme::astra(state.theme_mode);

    frame.render_widget(
        Block::default().style(theme.text_style()),
        area,
    );

    let regions = layout::regions::compute(
        area,
        state.left_rail_visible,
        state.left_rail_width,
    );

    command_ribbon::render(frame, regions.ribbon, state, &theme);

    if let Some(rail) = regions.left_rail {
        left_rail::render(frame, rail, state, &theme);
    }

    transcript::render(frame, regions.transcript, state, &theme);
    composer::render(frame, regions.composer, state, &theme);
    status_strip::render(frame, regions.status, state, &theme);

    if let Some(dialog) = &state.dialog {
        let area = centered_rect(70, 70, frame.area());
        frame.render_widget(Clear, area);
        dialog::render(frame, area, dialog, &theme);
    }

    if let Some(toast) = &state.toast {
        toast::render(frame, frame.area(), toast, &theme);
    }
}

        height: area.height.saturating_sub(2),
    };

    let chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(1),
            Constraint::Min(1),
            Constraint::Length(1),
        ])
        .split(inner);

    let mode = match state.composer.mode {
        ComposerMode::Ask => "ASK",
        ComposerMode::Shell => "SHELL",
        ComposerMode::Plan => "PLAN",
        ComposerMode::Build => "BUILD",
    };

    frame.render_widget(
        Paragraph::new(Line::from(vec![
            Span::styled(mode, Style::default().fg(theme.accent).add_modifier(Modifier::BOLD)),
            Span::styled(" · ", Style::default().fg(theme.muted)),
            Span::styled("Describe the change, attach context, or run a command", Style::default().fg(theme.muted)),
        ])),
        chunks[0],
    );

    let input = if state.composer.input.is_empty() {
        "Describe the change…"
    } else {
        &state.composer.input
    };

    let input_style = if state.composer.input.is_empty() {
        Style::default().fg(theme.muted).bg(theme.panel)
    } else {
        Style::default().fg(theme.text).bg(theme.panel)
    };

    frame.render_widget(
        Paragraph::new(input)
            .style(input_style)
            .wrap(Wrap { trim: false }),
        chunks[1],
    );

    frame.render_widget(
        Paragraph::new(Line::from(vec![
            Span::styled("Enter", Style::default().fg(theme.secondary).add_modifier(Modifier::BOLD)),
            Span::styled(" send  ", Style::default().fg(theme.muted)),
            Span::styled("Esc", Style::default().fg(theme.secondary).add_modifier(Modifier::BOLD)),
            Span::styled(" clear  ", Style::default().fg(theme.muted)),
            Span::styled("Ctrl+.", Style::default().fg(theme.accent).add_modifier(Modifier::BOLD)),
            Span::styled(" theme", Style::default().fg(theme.muted)),
        ])),
        chunks[2],
    );
}
```

This becomes the signature bottom surface of the product.

## 17. HTTP API bridge

```rust
// src/client/api.rs

use serde::Serialize;

#[derive(Clone)]
pub struct ApiClient {
    base_url: String,
    http: reqwest::Client,
}

impl ApiClient {
    pub fn new(base_url: impl Into<String>) -> Self {
        Self {
            base_url: base_url.into(),
            http: reqwest::Client::new(),
        }
    }

    pub async fn submit_prompt(
        &self,
        session_id: &str,
        text: &str,
    ) -> color_eyre::Result<()> {
        #[derive(Serialize)]
        struct Body<'a> {
            text: &'a str,
        }

        let url = format!("{}/session/{}/prompt", self.base_url, session_id);

        self.http
            .post(url)
            .json(&Body { text })
            .send()
            .await?
            .error_for_status()?;

        Ok(())
    }

    pub async fn abort_session(&self, session_id: &str) -> color_eyre::Result<()> {
        let url = format!("{}/session/{}/abort", self.base_url, session_id);

        self.http
            .post(url)
            .send()
            .await?
            .error_for_status()?;

        Ok(())
    }
}
```

This maps cleanly to the documented backend surface.

## 18. What Ratatui lets you do better than the current frontend

Ratatui gives you a more disciplined path to a truly premium terminal app:

| Area             | Existing OpenTUI/Solid approach    | Ratatui approach                             |
| ---------------- | ---------------------------------- | -------------------------------------------- |
| Rendering        | Retained/reactive TS components    | Immediate-mode Rust frame render             |
| Layout           | JSX-style boxes                    | Explicit `Rect` layout engine                |
| Performance      | JS event/render layer              | Native Rust draw loop                        |
| Theming          | JSON → RGBA theme resolver         | Strongly typed theme structs, optional serde |
| Input            | OpenTUI keyboard hooks             | Unified event reducer                        |
| Backend stream   | SDK/SSE provider                   | Direct async SSE task                        |
| Widgets          | Custom JSX components              | Native widgets + custom `Widget` impls       |
| Distinctiveness  | Still near inherited OpenCode tree | Entirely new app shell                       |
| Recognition risk | Higher                             | Much lower                                   |

Ratatui 0.30’s modularization also means application authors can still use the main `ratatui` crate while widget authors can depend on `ratatui-core` for lower dependency weight and API stability. That matters if you later build a public plugin/widget ecosystem around your app. ([Docs.rs][4])

## 20. How to preserve plugins

The current TS plugin system will not automatically carry over to Ratatui. I recommend splitting plugin support into two categories:

### Phase 1: backend-driven plugin data

Plugins can provide data through backend events/API. Ratatui renders known panel types:

```rust
pub enum RailPanel {
    Context(ContextPanel),
    Files(FilesPanel),
    Mcp(McpPanel),
    Lsp(LspPanel),
    Tasks(TaskPanel),
    CustomJson(CustomPanel),
}
```

### Phase 2: Rust widget plugins

Later, expose a plugin interface around `ratatui-core` widget traits. Since Ratatui 0.30 modularized core types into `ratatui-core`, third-party widget authors can target the core crate instead of pulling the full app/widget/backend stack. ([Docs.rs][4])

## 22. Final Ratatui recommendation

I would revise the project goal like this:

> The current OpenTUI/Solid frontend should be treated as the compatibility client. The flagship should become a Rust-native Ratatui frontend with a left mission rail, command ribbon, transcript canvas, composer deck, and first-class dark/white theme switching.

That gives you three huge wins:

1. **A completely different identity** from OpenCode.
2. **A true Rust TUI foundation** instead of a TypeScript terminal layer.
3. **A more powerful long-term rendering model** for dashboards, streaming state, widgets, charts, gauges, and custom app panels.

The Ratatui path is more work than restyling the existing code, but it is the path I would choose for the “world’s greatest Rust TUI coding app.”
