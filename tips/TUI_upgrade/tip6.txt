Yes — that is the right move **if the goal is a world-class Rust-native TUI that no longer feels like OpenCode**.

The strongest architecture is:

```text
Keep the existing backend/API contract.
Replace the current OpenTUI/SolidJS frontend with a first-class Rust Ratatui frontend.
```

The archive’s current frontend is TSX/OpenTUI, but Ratatui is a much better fit for the product ambition: Rust-native, fast, highly customizable, testable, and capable of building a completely distinct visual language. Ratatui is a Rust library for fast, lightweight, rich terminal interfaces, and its current site shows version `0.30.0`. It provides flexible widgets/layout/styling rather than forcing a fixed app framework, which is exactly what you want for a unique design. ([Ratatui][1])

# Ratatui Addendum: Recommended Rust-Native TUI Direction

## 1. Strategic Recommendation

I would **not** try to “sprinkle Ratatui” into the current TSX/OpenTUI app.

Instead, build a new Rust frontend crate:

```text
crates/jekko-tui/
```

or:

```text
packages/jekko-rs/tui/
```

This new frontend should consume the same backend events and command APIs described in `TUI_BACKEND_API.md`, but render everything through Ratatui.

The existing TSX TUI can remain temporarily as a fallback during migration, but the flagship experience should become the Rust/Ratatui app.

Why this is the better path:

* It gives the app a completely new visual identity.
* It breaks the OpenCode lineage at the UI framework level.
* It enables snapshot testing through Ratatui’s test backend.
* It gives you complete control over layout, input, animation cadence, rendering, overlays, and state.
* It lets you build a serious terminal “coding cockpit” rather than another chat panel.

Ratatui expects the application to own the event loop, app state, and redraw process, while Ratatui handles the terminal rendering primitives. That extra control is an advantage here because you want a distinctive, premium design rather than framework-default UI behavior. ([Ratatui][2])

# 4. Core Ratatui Architecture

Ratatui works best with an immediate-mode render loop:

```text
Backend event stream
        ↓
Internal app event
        ↓
State reducer
        ↓
Render full frame
        ↓
Terminal buffer diff
```

Recommended event enum:

```rust
#[derive(Debug, Clone)]
pub enum AppEvent {
    Init,
    Tick,
    Render,
    Quit,

    Key(crossterm::event::KeyEvent),
    Mouse(crossterm::event::MouseEvent),
    Paste(String),
    Resize(u16, u16),
    FocusGained,
    FocusLost,

    Backend(BackendEvent),
    Command(Command),
}
```

Ratatui’s own async tutorial recommends adding explicit `Tick` and `Render` events and using an async event model so the UI can render even while waiting on keyboard input or backend events. ([Ratatui][4])

Recommended event loop:

```rust
pub struct Tui {
    terminal: ratatui::Terminal<ratatui::backend::CrosstermBackend<std::io::Stderr>>,
    event_rx: tokio::sync::mpsc::UnboundedReceiver<AppEvent>,
    event_tx: tokio::sync::mpsc::UnboundedSender<AppEvent>,
}

impl Tui {
    pub async fn run(mut self, mut app: App) -> color_eyre::Result<()> {
        self.event_tx.send(AppEvent::Init)?;

        while !app.should_quit {
            tokio::select! {
                Some(event) = self.event_rx.recv() => {
                    app.handle_event(event)?;

                    self.terminal.draw(|frame| {
                        crate::views::root::render(frame, &mut app);
                    })?;
                }
            }
        }

        Ok(())
    }
}
```

Render only from a single UI thread. Backend SSE, timers, keyboard input, and command execution should send messages into the app event channel.

# 5. Preserve the Backend Contract

Do **not** rewrite the backend first. Keep the current backend and replace the visual layer.

The Rust Ratatui frontend should call the same conceptual backend APIs:

```text
GET    /event
POST   /tui/execute-command
POST   /tui/select-session
POST   /tui/open-themes
POST   /session/...
POST   /permission/reply
```

Recommended Rust API module:

```rust
pub struct BackendClient {
    base_url: String,
    client: reqwest::Client,
}

impl BackendClient {
    pub fn new(base_url: impl Into<String>) -> Self {
        Self {
            base_url: base_url.into(),
            client: reqwest::Client::new(),
        }
    }

    pub async fn execute_command(&self, command: &str) -> color_eyre::Result<()> {
        let url = format!("{}/tui/execute-command", self.base_url);
        self.client
            .post(url)
            .json(&serde_json::json!({ "command": command }))
            .send()
            .await?
            .error_for_status()?;
        Ok(())
    }

    pub async fn select_session(&self, session_id: &str) -> color_eyre::Result<()> {
        let url = format!("{}/tui/select-session", self.base_url);
        self.client
            .post(url)
            .json(&serde_json::json!({ "sessionID": session_id }))
            .send()
            .await?
            .error_for_status()?;
        Ok(())
    }
}
```

The key idea: backend behavior stays familiar; UI behavior becomes entirely new.

```rust
pub enum ThemeMode {
    Dark,
    White,
}

impl ThemeMode {
    pub fn toggle(self) -> Self {
        match self {
            Self::Dark => Self::White,
            Self::White => Self::Dark,
        }
    }
}
```

Key handling:

```rust
pub struct KeyChordState {
    leader_active: bool,
}

impl App {
    pub fn handle_key(&mut self, key: KeyEvent) -> color_eyre::Result<()> {
        use crossterm::event::{KeyCode, KeyModifiers};

        if key.code == KeyCode::Char('x') && key.modifiers.contains(KeyModifiers::CONTROL) {
            self.key_chord.leader_active = true;
            return Ok(());
        }

        if self.key_chord.leader_active {
            self.key_chord.leader_active = false;

            match key.code {
                KeyCode::Char('w') | KeyCode::Char('W') => {
                    self.theme.mode = self.theme.mode.toggle();
                    self.toast.info(match self.theme.mode {
                        ThemeMode::Dark => "Dark mode enabled",
                        ThemeMode::White => "White mode enabled",
                    });
                    return Ok(());
                }
                KeyCode::Char('p') | KeyCode::Char('P') => {
                    self.ui.open_command_deck();
                    return Ok(());
                }
                _ => {}
            }
        }

        Ok(())
    }
}
```

# 15. Final Recommendation

The best product decision is:

```text
Use Ratatui for the next-generation TUI.
Keep the backend.
Do not preserve the OpenCode layout language.
Ship Atlas/White and Eclipse/Dark from day one.
Move the coding panel to the left.
Make Ctrl+X W the White/Dark shortcut.
```

This gives you a Rust-native terminal app with a truly different identity: a premium coding cockpit, not a fork skin.
