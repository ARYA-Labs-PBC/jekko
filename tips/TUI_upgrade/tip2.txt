# Updated Engineering Report: Use Ratatui as the End-State TUI

Yes — **if the goal is a world-class coding TUI with a completely different look and feel, Ratatui should become the preferred end-state**, not just an inspiration layer.

The code you uploaded is currently **TypeScript/TSX using OpenTUI + SolidJS**, even though the project is described as Rust TUI code. That matters because Ratatui cannot be directly imported into the existing TSX rendering tree. To fully leverage Ratatui, the right move is to keep the existing backend/API worker model and replace the visible TUI frontend with a new Rust/Ratatui binary.

Ratatui is a Rust library for building fast, lightweight, rich terminal interfaces, and its current documentation emphasizes rich widgets, dynamic responsive layouts, dashboards, and modern terminal experiences. It also supports tables, charts, gauges, lists, paragraphs, scrollbars, blocks, sparklines, tabs, canvas drawing, and stateful widgets — exactly the toolkit needed for a premium coding cockpit. ([Ratatui][1])

My updated recommendation:

> **Do not merely restyle the current OpenTUI/Solid app. Use it as a behavioral reference and build a new Rust/Ratatui frontend that talks to the same backend API and SSE event stream.**

That gives you the biggest possible visual break from OpenCode while also using the strongest Rust-native TUI stack.

## 2. Why Ratatui Changes the Architecture

The uploaded TUI uses this chain:

```text
thread.ts
  → worker.ts starts backend HTTP server
  → app.tsx creates OpenTUI renderer
  → SolidJS provider tree
  → TSX visual components
```

With Ratatui, the better architecture becomes:

```text
thread.ts
  → worker.ts starts same backend HTTP server
  → spawn Rust/Ratatui frontend process
  → Rust TUI connects to backend URL
  → Rust TUI subscribes to /event SSE
  → Ratatui renders entire interface
```

This means the backend API documented in `TUI_BACKEND_API.md` remains extremely valuable. The frontend can continue using:

```text
GET  /event
GET  /session
POST /session/:id/prompt
POST /session/:id/shell
GET  /vcs/diff
GET  /file/status
GET  /mcp/status
GET  /lsp
POST /permission/:id/reply
POST /question/:id/reply
POST /tui/show-toast
```

Ratatui supports multiple terminal backends, including Crossterm by default, plus Termion, Termwiz, and a TestBackend for UI tests. It also supports raw mode, alternate screen, and mouse capture through backends. ([Ratatui][2])

Important: Ratatui is a library, not an app framework. It gives you widgets and rendering primitives but leaves the event loop, state architecture, and data flow under your control. That is a good thing here because this app already has a strong backend/event model. ([Ratatui][3])

```rust
// crates/code-command-tui/src/main.rs

mod action;
mod app;
mod backend;
mod config;
mod error;
mod input;
mod layout;
mod state;
mod theme;
mod ui;

use anyhow::Result;
use app::App;

fn main() -> Result<()> {
    let config = config::Config::from_env_and_args()?;

    ratatui::run(|terminal| {
        let runtime = tokio::runtime::Builder::new_multi_thread()
            .enable_all()
            .build()?;

        runtime.block_on(async {
            let mut app = App::new(config).await?;
            app.run(terminal).await
        })
    })?;

    Ok(())
}
```

This keeps the nice Ratatui initialization/restoration path while still letting the app use async HTTP/SSE networking.

## 7. Main App Loop

```rust
// crates/code-command-tui/src/app.rs

use anyhow::Result;
use crossterm::event::{self, Event as CrosstermEvent};
use ratatui::DefaultTerminal;
use std::time::{Duration, Instant};

use crate::{
    action::Action,
    backend::{client::BackendClient, events::BackendEventStream},
    input,
    state::AppState,
    ui,
};

pub struct App {
    state: AppState,
    client: BackendClient,
    events: BackendEventStream,
    tick_rate: Duration,
}

impl App {
    pub async fn new(config: crate::config::Config) -> Result<Self> {
        let client = BackendClient::new(config.backend_url.clone(), config.headers.clone());
        let events = BackendEventStream::connect(config.backend_url, config.headers).await?;

        let mut state = AppState::new(config);
        state.bootstrap(&client).await?;

        Ok(Self {
            state,
            client,
            events,
            tick_rate: Duration::from_millis(33),
        })
    }

    pub async fn run(&mut self, terminal: &mut DefaultTerminal) -> Result<()> {
        let mut last_tick = Instant::now();

        loop {
            terminal.draw(|frame| ui::render(frame, &mut self.state))?;

            while let Some(event) = self.events.try_next()? {
                self.state.apply_backend_event(event);
            }

            let timeout = self
                .tick_rate
                .saturating_sub(last_tick.elapsed());

            if event::poll(timeout)? {
                if let CrosstermEvent::Key(key) = event::read()? {
                    let action = input::map_key(key, &self.state);
                    if self.handle_action(action).await? {
                        break;
                    }
                }
            }

            if last_tick.elapsed() >= self.tick_rate {
                self.state.tick();
                last_tick = Instant::now();
            }
        }

        Ok(())
    }

    async fn handle_action(&mut self, action: Action) -> Result<bool> {
        match action {
            Action::Quit => Ok(true),
            Action::SubmitPrompt => {
                let prompt = self.state.composer.take_text();
                self.client.submit_prompt(self.state.current_session_id(), prompt).await?;
                Ok(false)
            }
            Action::OpenCommandPalette => {
                self.state.dialogs.open_command_palette();
                Ok(false)
            }
            Action::ToggleLeftPanel => {
                self.state.layout.toggle_left_panel();
                Ok(false)
            }
            Action::ComposerInput(ch) => {
                self.state.composer.push(ch);
                Ok(false)
            }
            Action::Noop => Ok(false),
        }
    }
}
```

Ratatui uses a draw model where all widgets should be rendered during the draw call rather than scattering multiple draw calls through the loop. Its FAQ specifically warns against multiple consecutive `terminal.draw()` calls for separate widgets because Ratatui uses double-buffer rendering. ([Ratatui][3])

## 8. Layout Engine: Left Panel First

Ratatui’s `Layout` supports dynamic splits using constraints such as length, min, max, ratio, and percentage, and layouts can be nested. That directly solves the current hardcoded `42` column sidebar and `> 120` breakpoint issues in the uploaded TSX code. ([Ratatui][5])

```rust
// crates/code-command-tui/src/layout.rs

use ratatui::layout::{Constraint, Direction, Layout, Rect};

#[derive(Debug, Clone, Copy)]
pub struct ShellAreas {
    pub top: Rect,
    pub left: Option<Rect>,
    pub main: Rect,
    pub composer: Rect,
}

pub const LEFT_PANEL_WIDTH: u16 = 38;
pub const WIDE_BREAKPOINT: u16 = 118;
pub const TOP_RIBBON_HEIGHT: u16 = 3;
pub const COMPOSER_HEIGHT: u16 = 8;

pub fn shell(area: Rect, left_panel_visible: bool) -> ShellAreas {
    let vertical = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(TOP_RIBBON_HEIGHT),
            Constraint::Min(8),
            Constraint::Length(COMPOSER_HEIGHT),
        ])
        .split(area);

    let top = vertical[0];
    let body = vertical[1];
    let composer = vertical[2];

    let wide = area.width >= WIDE_BREAKPOINT;
    let show_left = left_panel_visible && wide;

    if show_left {
        let horizontal = Layout::default()
            .direction(Direction::Horizontal)
            .constraints([
                Constraint::Length(LEFT_PANEL_WIDTH),
                Constraint::Min(30),
            ])
            .split(body);

        ShellAreas {
            top,
            left: Some(horizontal[0]),
            main: horizontal[1],
            composer,
        }
    } else {
        ShellAreas {
            top,
            left: None,
            main: body,
            composer,
        }
    }
}
```

// crates/code-command-tui/src/ui/left_panel.rs

use ratatui::{
    layout::{Constraint, Direction, Layout, Rect},
    style::{Modifier, Style},
    text::{Line, Span},
    widgets::{Block, Borders, Gauge, List, ListItem, Paragraph},
    Frame,
};

use crate::state::AppState;

pub fn render(frame: &mut Frame, area: Rect, state: &mut AppState, overlay: bool) {
    let theme = &state.theme;

    let block = Block::default()
        .borders(if overlay { Borders::ALL } else { Borders::RIGHT })
        .border_style(Style::default().fg(if overlay {
            theme.border_active
        } else {
            theme.border_subtle
        }))
        .style(Style::default().bg(theme.panel));

    frame.render_widget(block, area);

    let inner = Rect {
        x: area.x + 1,
        y: area.y + 1,
        width: area.width.saturating_sub(2),
        height: area.height.saturating_sub(2),
    };

    let sections = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(5),
            Constraint::Length(7),
            Constraint::Length(7),
            Constraint::Min(8),
            Constraint::Length(3),
        ])
        .split(inner);

    render_workspace(frame, sections[0], state);
    render_context_meter(frame, sections[1], state);
    render_files(frame, sections[2], state);
    render_services(frame, sections[3], state);
    render_panel_footer(frame, sections[4], state);
}

fn render_workspace(frame: &mut Frame, area: Rect, state: &AppState) {
    let theme = &state.theme;

    let title = Line::from(vec![
        Span::styled("◆ ", Style::default().fg(theme.accent)),
        Span::styled("MISSION CONTROL", theme.title()),
    ]);

    let body = vec![
        title,
        Line::from(Span::styled(state.workspace.name.as_str(), Style::default().fg(theme.text))),
        Line::from(Span::styled(state.workspace.path.as_str(), Style::default().fg(theme.muted))),
    ];

    frame.render_widget(Paragraph::new(body), area);
}

fn render_context_meter(frame: &mut Frame, area: Rect, state: &AppState) {
    let theme = &state.theme;

    let block = Block::default()
        .borders(Borders::ALL)
        .title(" Context ")
        .border_style(Style::default().fg(theme.border_subtle));

    let ratio = state.usage.context_ratio();

    let gauge = Gauge::default()
        .block(block)
        .gauge_style(Style::default().fg(theme.primary).bg(theme.element))
        .ratio(ratio)
        .label(format!("{:.0}% used", ratio * 100.0));

    frame.render_widget(gauge, area);
}

fn render_files(frame: &mut Frame, area: Rect, state: &AppState) {
    let theme = &state.theme;

    let items = state
        .files
        .changed
        .iter()
        .take(5)
        .map(|file| {
            ListItem::new(Line::from(vec![
                Span::styled("◇ ", Style::default().fg(theme.secondary)),
                Span::styled(file.name.as_str(), Style::default().fg(theme.text)),
                Span::styled(
                    format!("  +{} -{}", file.additions, file.deletions),
                    Style::default().fg(theme.muted),
                ),
            ]))
        })
        .collect::<Vec<_>>();

    let list = List::new(items)
        .block(
            Block::default()
                .borders(Borders::ALL)
                .title(" Files ")
                .border_style(Style::default().fg(theme.border_subtle)),
        )
        .style(Style::default().bg(theme.panel));

    frame.render_widget(list, area);
}

fn render_services(frame: &mut Frame, area: Rect, state: &AppState) {
    let theme = &state.theme;

    let rows = vec![
        ListItem::new(status_line("LSP", state.services.lsp_count, theme.success, theme)),
        ListItem::new(status_line("MCP", state.services.mcp_count, theme.info, theme)),
        ListItem::new(status_line("Tasks", state.tasks.pending_count, theme.warning, theme)),
    ];

    let list = List::new(rows).block(
        Block::default()
            .borders(Borders::ALL)
            .title(" Systems ")
            .border_style(Style::default().fg(theme.border_subtle)),
    );

    frame.render_widget(list, area);
}

fn status_line(label: &str, count: usize, color: ratatui::style::Color, theme: &crate::theme::Theme) -> Line<'static> {
    Line::from(vec![
        Span::styled("● ", Style::default().fg(color)),
        Span::styled(label.to_owned(), Style::default().fg(theme.text).add_modifier(Modifier::BOLD)),
        Span::styled(format!(" {}", count), Style::default().fg(theme.muted)),
    ])
}

fn render_panel_footer(frame: &mut Frame, area: Rect, state: &AppState) {
    let theme = &state.theme;

    let line = Line::from(vec![
        Span::styled("● ", Style::default().fg(theme.success)),
        Span::styled("Ready", Style::default().fg(theme.text)),
        Span::styled("  /status", Style::default().fg(theme.muted)),
    ]);

    frame.render_widget(Paragraph::new(line), area);
}

### Dialog rendering pattern

Use Ratatui’s `Clear` widget for overlays, then render a centered block. The official widgets list includes `Clear`, which is specifically useful for rendering over previously drawn widgets. ([Ratatui][6])

```rust
// crates/code-command-tui/src/ui/dialogs.rs

use ratatui::{
    layout::{Alignment, Constraint, Direction, Layout, Rect},
    style::{Modifier, Style},
    widgets::{Block, Borders, Clear, List, ListItem, Paragraph},
    Frame,
};

use crate::state::AppState;

pub fn render(frame: &mut Frame, state: &mut AppState) {
    if state.dialogs.command_palette_open {
        render_command_palette(frame, state);
    }
}

fn centered_rect(area: Rect, percent_x: u16, percent_y: u16) -> Rect {
    let vertical = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Percentage((100 - percent_y) / 2),
            Constraint::Percentage(percent_y),
            Constraint::Percentage((100 - percent_y) / 2),
        ])
        .split(area);

    Layout::default()
        .direction(Direction::Horizontal)
        .constraints([
            Constraint::Percentage((100 - percent_x) / 2),
            Constraint::Percentage(percent_x),
            Constraint::Percentage((100 - percent_x) / 2),
        ])
        .split(vertical[1])[1]
}

fn render_command_palette(frame: &mut Frame, state: &mut AppState) {
    let theme = &state.theme;
    let area = centered_rect(frame.area(), 64, 58);

    frame.render_widget(Clear, area);

    let block = Block::default()
        .borders(Borders::ALL)
        .title(" COMMAND CENTER ")
        .title_alignment(Alignment::Center)
        .title_style(Style::default().fg(theme.primary).add_modifier(Modifier::BOLD))
        .border_style(Style::default().fg(theme.border_active))
        .style(Style::default().bg(theme.panel));

    frame.render_widget(block, area);

    let inner = Rect {
        x: area.x + 2,
        y: area.y + 1,
        width: area.width.saturating_sub(4),
        height: area.height.saturating_sub(2),
    };

    let items = state
        .commands
        .filtered()
        .iter()
        .enumerate()
        .map(|(idx, cmd)| {
            let selected = idx == state.commands.selected;
            let marker = if selected { "◆ " } else { "  " };
            let style = if selected {
                Style::default().fg(theme.accent).bg(theme.element)
            } else {
                Style::default().fg(theme.text)
            };

            ListItem::new(format!("{marker}{:<28} {}", cmd.id, cmd.description)).style(style)
        })
        .collect::<Vec<_>>();

    frame.render_widget(List::new(items), inner);
}
```

## 17. Backend Client Sketch

The current backend API can remain stable. The Rust frontend needs a thin client:

```rust
// crates/code-command-tui/src/backend/client.rs

use anyhow::Result;
use reqwest::Client;
use serde::{Deserialize, Serialize};
use url::Url;

#[derive(Clone)]
pub struct BackendClient {
    http: Client,
    base: Url,
    headers: Vec<(String, String)>,
}

impl BackendClient {
    pub fn new(base: Url, headers: Vec<(String, String)>) -> Self {
        Self {
            http: Client::new(),
            base,
            headers,
        }
    }

    pub async fn list_sessions(&self) -> Result<Vec<Session>> {
        let url = self.base.join("/session")?;
        let res = self.http.get(url).send().await?;
        Ok(res.json().await?)
    }

    pub async fn submit_prompt(&self, session_id: &str, text: String) -> Result<()> {
        let url = self.base.join(&format!("/session/{session_id}/prompt"))?;

        self.http
            .post(url)
            .json(&SubmitPrompt { text })
            .send()
            .await?
            .error_for_status()?;

        Ok(())
    }
}

#[derive(Debug, Deserialize)]
pub struct Session {
    pub id: String,
    pub title: String,
}

#[derive(Debug, Serialize)]
struct SubmitPrompt {
    text: String,
}
```

### SSE events

```rust
// crates/code-command-tui/src/backend/events.rs

use anyhow::Result;
use futures_util::StreamExt;
use reqwest::Client;
use tokio::sync::mpsc;
use url::Url;

use crate::backend::models::BackendEvent;

pub struct BackendEventStream {
    rx: mpsc::UnboundedReceiver<BackendEvent>,
}

impl BackendEventStream {
    pub async fn connect(base: Url, _headers: Vec<(String, String)>) -> Result<Self> {
        let (tx, rx) = mpsc::unbounded_channel();
        let url = base.join("/event")?;

        tokio::spawn(async move {
            let client = Client::new();

            loop {
                let result = client.get(url.clone()).send().await;

                match result {
                    Ok(response) => {
                        let mut stream = response.bytes_stream();

                        while let Some(chunk) = stream.next().await {
                            let Ok(bytes) = chunk else { break };
                            // Parse Server-Sent Events here.
                            // For production, use a proper SSE parser.
                            if let Some(event) = parse_sse_chunk(&bytes) {
                                let _ = tx.send(event);
                            }
                        }
                    }
                    Err(_) => {
                        tokio::time::sleep(std::time::Duration::from_millis(500)).await;
                    }
                }
            }
        });

        Ok(Self { rx })
    }

    pub fn try_next(&mut self) -> Result<Option<BackendEvent>> {
        Ok(self.rx.try_recv().ok())
    }
}

fn parse_sse_chunk(_bytes: &[u8]) -> Option<BackendEvent> {
    None
}
```

For production, use a robust SSE parser and mirror the event schemas from `TUI_BACKEND_API.md`.

## 18. TS Launcher Changes

You do not need to throw away the current backend worker. Use it to launch the Rust frontend.

### `thread.ts`

Change from:

```text
thread.ts
  → app.tsx
```

to:

```text
thread.ts
  → worker.ts starts backend
  → spawn code-command-tui binary with --server-url
```

Example direction:

```ts
// packages/jekko/src/cli/cmd/tui/thread.ts

import { spawn } from "node:child_process"
import { fileURLToPath } from "node:url"

async function launchRatatui(input: {
  serverUrl: string
  directory: string
  headers?: Record<string, string>
}) {
  const bin = process.env.CODE_COMMAND_TUI_BIN ?? resolveBundledRatatuiBinary()

  const child = spawn(
    bin,
    [
      "--server-url",
      input.serverUrl,
      "--directory",
      input.directory,
    ],
    {
      stdio: "inherit",
      env: {
        ...process.env,
        CODE_COMMAND_TUI_HEADERS: JSON.stringify(input.headers ?? {}),
      },
    },
  )

  await new Promise<void>((resolve, reject) => {
    child.on("exit", (code) => {
      if (code === 0) resolve()
      else reject(new Error(`Ratatui frontend exited with code ${code}`))
    })
    child.on("error", reject)
  })
}

function resolveBundledRatatuiBinary() {
  // Resolve platform-specific installed binary.
  // Example:
  // dist/bin/code-command-tui-${process.platform}-${process.arch}
  return fileURLToPath(new URL("../../../../bin/code-command-tui", import.meta.url))
}
```

### `worker.ts`

Keep mostly unchanged. It still provides the backend HTTP API and SSE stream.

### `app.tsx`, `app-view.tsx`

Retire from production path. Keep only as fallback:

```ts
if (process.env.USE_LEGACY_OPENTUI === "1") {
  await tui(...)
} else {
  await launchRatatui(...)
}
```

## 20. Plugin System Warning

The current visual plugin system is SolidJS slot-based:

```text
plugin/slots.tsx
plugin/api.tsx
feature-plugins/*
```

A Rust/Ratatui frontend cannot directly render SolidJS plugin components.

Recommended approach:

### Phase 1

Do not support arbitrary visual JS plugins in the new Ratatui frontend. Rebuild all internal feature plugins as native Rust panels.

### Phase 2

Add a **terminal card DSL** for external plugin UI:

```json
{
  "type": "panel_card",
  "title": "Build Health",
  "tone": "success",
  "rows": [
    { "label": "Tests", "value": "128 passing" },
    { "label": "Coverage", "value": "94.2%" }
  ]
}
```

Then Ratatui renders plugin output safely as:

```text
┌ Build Health ───────────────┐
│ Tests        128 passing    │
│ Coverage     94.2%          │
└─────────────────────────────┘
```

This avoids embedding JS UI rendering into Rust and gives you consistent visual quality.

## 21. Fonts

Ratatui cannot control terminal font size or force a user’s font. The official FAQ states that Ratatui renders based on the terminal’s size/capabilities and that users must adjust font settings in their terminal emulator. ([Ratatui][3])

So the app should:

1. Recommend fonts in docs/settings.
2. Avoid hard dependency on Nerd Font glyphs.
3. Provide glyph modes.

Recommended config:

```rust
#[derive(Debug, Clone, Copy)]
pub enum GlyphMode {
    Unicode,
    Ascii,
    Nerd,
}
```

Recommended docs:

```text
Recommended terminal font:
- Monaspace Neon
- JetBrains Mono
- Iosevka Term
- Geist Mono
- Berkeley Mono
```

Default glyph mode:

```text
unicode
```

Fallbacks:

```text
◆ → *
◇ → -
╭╮╰╯ → +---+
⌃P → Ctrl-P
```
