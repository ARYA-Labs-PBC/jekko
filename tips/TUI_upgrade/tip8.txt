# Ratatui-First Addendum — Best Path for the “World’s Greatest Rust TUI”

Yes — I would absolutely leverage Ratatui, and I would treat it as the **best long-term frontend direction**.

The important architectural point: the code I reviewed is currently an **OpenTUI + SolidJS TypeScript frontend**, even though it lives in a CLI/TUI layer. Ratatui cannot be sprinkled into that UI as a minor styling change. To get the full value of Ratatui, we should build a **new Rust Ratatui frontend** that preserves the existing backend/API contract from `TUI_BACKEND_API.md`.

That is actually excellent news. It gives you the freedom you asked for: not limited by the current design, not visually recognizable as OpenCode, and able to build a premium Rust-native TUI with a completely new silhouette.

Ratatui is a strong fit here because it provides immediate-mode rendering, dynamic constraint-based layouts, rich widgets such as tables, gauges, charts, sparklines, lists, and scrollbars, and custom widget support. Its docs also emphasize responsive layouts that adapt from small terminal panes to large screens, which maps perfectly to the left-rail + coding-canvas direction. ([Ratatui][1])

---

# Recommendation

Build a new crate:

```text
crates/apex-tui/
```

or, using your real product name:

```text
crates/<product>-tui/
```

This crate becomes the new Rust Ratatui frontend. The existing TypeScript TUI can remain temporarily as `legacy-opentui` during migration, but the production target should be the Ratatui frontend.

The backend does **not** need a large rewrite. The existing backend API already exposes sessions, messages, prompt submission, TUI commands, provider data, permissions, questions, MCP/LSP status, PTY endpoints, and SSE event streaming. That means the Rust frontend can become a first-class client over the same REST/SSE surface.

# Why Ratatui Changes the Design Strategy

The previous report recommended changing the TypeScript/OpenTUI layer. With Ratatui available, I would make the strategy bolder:

```text
Current:
OpenTUI/Solid frontend
        ↓
Backend HTTP/SSE API

Recommended:
Rust Ratatui frontend
        ↓
Same backend HTTP/SSE API
```

This gives you:

1. A truly Rust-native TUI.
2. A totally new visual identity.
3. Stronger layout control.
4. Better custom widgets.
5. More reliable snapshot testing.
6. Easier high-performance terminal rendering.
7. A clean break from recognizable OpenCode/OpenTUI structure.

Ratatui’s `Widget` and `StatefulWidget` traits are exactly what we want for premium surfaces like the left rail, message stream, command composer, permission cards, model picker, command palette, diff viewer, token gauges, and live activity dashboard. ([Ratatui][2])

# New Ratatui App Architecture

Use an action/event architecture:

```text
Terminal input
Backend SSE event
Timer tick
Remote TUI control event
        ↓
Action enum
        ↓
App::update(action)
        ↓
AppState
        ↓
Ratatui render(frame, &app)
```

Ratatui itself is not fundamentally an async library; its FAQ says the main Ratatui interface is `terminal.draw(|f| ui(f))`, while async is useful for the parts of the app that need it, such as input streams or backend work. That is exactly how this app should be built: keep rendering synchronous and deterministic, but feed it with async backend and input events through channels. ([Ratatui][5])

# `src/app.rs`

```rust
use crate::{
    action::Action,
    state::Store,
    theme::ThemeMode,
};

pub struct App {
    pub store: Store,
    pub should_quit: bool,
    pub needs_render: bool,
}

impl App {
    pub fn new(store: Store) -> Self {
        Self {
            store,
            should_quit: false,
            needs_render: true,
        }
    }

    pub fn update(&mut self, action: Action) {
        match action {
            Action::Quit => {
                self.should_quit = true;
            }

            Action::ToggleThemeMode => {
                self.store.theme.mode = match self.store.theme.mode {
                    ThemeMode::Dark => ThemeMode::Light,
                    ThemeMode::Light => ThemeMode::Dark,
                    ThemeMode::System => ThemeMode::Dark,
                };

                self.store.toast.info(format!(
                    "Theme mode: {}",
                    self.store.theme.mode.label()
                ));
            }

            Action::ToggleLeftRail => {
                self.store.layout.left_rail_open = !self.store.layout.left_rail_open;
            }

            Action::OpenCommandPalette => {
                self.store.dialogs.open_command_palette();
            }

            Action::PromptInput(input) => {
                self.store.prompt.input = input;
            }

            Action::PromptSubmit => {
                self.store.prompt.submit_requested = true;
            }

            Action::BackendEvent(event) => {
                self.store.apply_backend_event(event);
            }

            Action::Toast(toast) => {
                self.store.toast.push(toast);
            }

            _ => {}
        }

        self.needs_render = true;
    }
}
```

# `src/terminal.rs`

```rust
use color_eyre::Result;
use crossterm::event::{Event as CrosstermEvent, EventStream};
use futures_util::StreamExt;
use ratatui::DefaultTerminal;
use tokio::sync::mpsc;

use crate::action::Action;

pub async fn run_terminal(
    mut terminal: DefaultTerminal,
    mut app: crate::app::App,
    tx: mpsc::UnboundedSender<Action>,
    mut rx: mpsc::UnboundedReceiver<Action>,
) -> Result<()> {
    spawn_input_task(tx.clone());

    let ticker = tokio::spawn(async move {
        let mut interval = tokio::time::interval(std::time::Duration::from_millis(33));
        loop {
            interval.tick().await;
            let _ = tx.send(Action::Tick);
        }
    });

    while !app.should_quit {
        if app.needs_render {
            terminal.draw(|frame| crate::ui::root::render(frame, &mut app))?;
            app.needs_render = false;
        }

        if let Some(action) = rx.recv().await {
            app.update(action);
        }
    }

    ticker.abort();
    Ok(())
}

fn spawn_input_task(tx: mpsc::UnboundedSender<Action>) {
    tokio::spawn(async move {
        let mut reader = EventStream::new();

        while let Some(event) = reader.next().await {
            match event {
                Ok(CrosstermEvent::Key(key)) => {
                    let _ = tx.send(Action::Key(key));
                }
                Ok(CrosstermEvent::Resize(_, _)) => {
                    let _ = tx.send(Action::Render);
                }
                _ => {}
            }
        }
    });
}
```

```rust
mod action;
mod app;
mod backend;
mod config;
mod keymap;

mod theme;
mod ui;

    color_eyre::install()?;

    let config = config::Config::load()?;
    let store = state::Store::from_config(config).await?;
    let app = app::App::new(store);

    let terminal = ratatui::init();

    let (tx, rx) = mpsc::unbounded_channel();

    backend::events::spawn_event_stream(app.store.backend.clone(), tx.clone());
    backend::control::spawn_control_loop(app.store.backend.clone(), tx.clone());

    let result = terminal::run_terminal(terminal, app, tx, rx).await;

    ratatui::restore();

    result
}
```

# New Layout: Left Rail + Coding Canvas

Ratatui’s layout model uses rectangles and constraints such as `Length`, `Min`, `Max`, `Ratio`, and `Percentage`. That is perfect for the target silhouette: fixed left rail, flexible canvas, fixed composer, responsive collapse on narrow terminals. ([Ratatui][6])

## `src/ui/root.rs`

```rust
use ratatui::{
    prelude::*,
    widgets::Clear,
};

use crate::app::App;

pub fn render(frame: &mut Frame, app: &mut App) {
    let area = frame.area();
    let palette = app.store.theme.palette();

    frame.render_widget(
        Block::new().bg(palette.background),
        area,
    );

    let [header, body, composer] = Layout::vertical([
        Constraint::Length(1),
        Constraint::Min(8),
        Constraint::Length(app.store.layout.composer_height()),
    ])
    .areas(area);

    crate::ui::header::render(frame, app, header);

    render_body(frame, app, body);

    crate::ui::composer::render(frame, app, composer);

    if let Some(dialog) = app.store.dialogs.active() {
        crate::ui::dialogs::render(frame, app, dialog, area);
    }

    if let Some(toast) = app.store.toast.current() {
        crate::ui::toast::render(frame, app, toast, area);
    }
}

fn render_body(frame: &mut Frame, app: &mut App, area: Rect) {
    let wide = area.width >= 110;
    let rail_visible = app.store.layout.left_rail_open || wide;

    if rail_visible && wide {
        let [rail, canvas] = Layout::horizontal([
            Constraint::Length(app.store.layout.left_rail_width),
            Constraint::Min(40),
        ])
        .areas(area);

        crate::ui::left_rail::render(frame, app, rail);
        crate::ui::session::view::render(frame, app, canvas);
        return;
    }

    crate::ui::session::view::render(frame, app, area);

    if app.store.layout.left_rail_open {
        let drawer_width = area.width.min(app.store.layout.left_rail_width);
        let drawer = Rect {
            x: area.x,
            y: area.y,
            width: drawer_width,
            height: area.height,
        };

        frame.render_widget(Clear, drawer);
        crate::ui::left_rail::render(frame, app, drawer);
    }
}
```

# Dark/Light Shortcut

Use:

```text
Ctrl+Shift+L
```

with fallback:

```text
Ctrl+Alt+L
```

Not every terminal reports shifted control keys consistently, so keep the fallback and also expose `/mode`, `/dark`, `/light`, and command palette search.

## `src/keymap.rs`

```rust
use crossterm::event::{KeyCode, KeyEvent, KeyModifiers};

use crate::action::Action;

pub fn map_key(key: KeyEvent) -> Option<Action> {
    match (key.code, key.modifiers) {
        (KeyCode::Char('c'), mods) if mods.contains(KeyModifiers::CONTROL) => {
            Some(Action::Quit)
        }

        (KeyCode::Char('k'), mods) if mods.contains(KeyModifiers::CONTROL) => {
            Some(Action::OpenCommandPalette)
        }

        (KeyCode::Char('b'), mods) if mods.contains(KeyModifiers::CONTROL) => {
            Some(Action::ToggleLeftRail)
        }

        (KeyCode::Char('l'), mods)
            if mods.contains(KeyModifiers::CONTROL)
                && mods.contains(KeyModifiers::SHIFT) =>
        {
            Some(Action::ToggleThemeMode)
        }

        (KeyCode::Char('l'), mods)
            if mods.contains(KeyModifiers::CONTROL)
                && mods.contains(KeyModifiers::ALT) =>
        {
            Some(Action::ToggleThemeMode)
        }

        _ => Some(Action::Key(key)),
    }
}
```

# Diff Viewer

Use muted gutters, strong semantic colors, and avoid looking like the original OpenCode diff blocks.

## `src/ui/session/diff_view.rs`

```rust
use ratatui::{prelude::*, widgets::Paragraph};

use crate::theme::Palette;

pub struct DiffView<'a> {
    pub lines: &'a [DiffLine],
    pub palette: Palette,
}

pub struct DiffLine {
    pub kind: DiffKind,
    pub old_no: Option<u32>,
    pub new_no: Option<u32>,
    pub text: String,
}

pub enum DiffKind {
    Added,
    Removed,
    Context,
    Hunk,
}

impl<'a> Widget for DiffView<'a> {
    fn render(self, area: Rect, buf: &mut Buffer) {
        let lines = self.lines.iter().map(|line| {
            let style = match line.kind {
                DiffKind::Added => Style::default().fg(self.palette.success),
                DiffKind::Removed => Style::default().fg(self.palette.error),
                DiffKind::Context => Style::default().fg(self.palette.text),
                DiffKind::Hunk => Style::default().fg(self.palette.secondary),
            };

            let prefix = match line.kind {
                DiffKind::Added => "+",
                DiffKind::Removed => "-",
                DiffKind::Context => " ",
                DiffKind::Hunk => "@",
            };

            Line::from(vec![
                Span::styled(
                    format!("{:>4} {:>4} ", opt_no(line.old_no), opt_no(line.new_no)),
                    Style::default().fg(self.palette.muted),
                ),
                Span::styled(prefix, style),
                Span::raw(" "),
                Span::styled(line.text.clone(), style),
            ])
        });

        Paragraph::new(lines.collect::<Vec<_>>())
            .style(Style::default().bg(self.palette.background))
            .render(area, buf);
    }
}

fn opt_no(value: Option<u32>) -> String {
    value.map(|v| v.to_string()).unwrap_or_else(|| " ".into())
}
```

Ratatui renders into a buffer, and the docs explain that cells can retain symbols/styles if an overlay widget does not clear the region first. For popups, use `Clear` before drawing the modal or drawer to prevent visual bleed-through. Ratatui’s rendering docs also explain the cell/buffer model, which is useful for custom overlays and precise styling. ([Ratatui][7])

```rust
use ratatui::{
    prelude::*,

pub fn centered_rect(area: Rect, percent_x: u16, percent_y: u16) -> Rect {
    let vertical = Layout::vertical([
        Constraint::Percentage((100 - percent_y) / 2),
        Constraint::Percentage(percent_y),
        Constraint::Percentage((100 - percent_y) / 2),
    ])
    .split(area);

    Layout::horizontal([
        Constraint::Percentage((100 - percent_x) / 2),
        Constraint::Percentage(percent_x),
        Constraint::Percentage((100 - percent_x) / 2),
    ])
    .split(vertical[1])[1]
}

        .style(Style::default().bg(palette.panel).fg(palette.text));

    let inner = block.inner(area);
    frame.render_widget(block, area);
    inner
}
```

# Backend Client

The Rust frontend should consume the existing API from `TUI_BACKEND_API.md`.

## `src/backend/client.rs`

```rust
use color_eyre::Result;
use reqwest::Client;
use serde::{Deserialize, Serialize};

#[derive(Clone)]
pub struct BackendClient {
    http: Client,
    base_url: String,
}

impl BackendClient {
    pub fn new(base_url: impl Into<String>) -> Self {
        Self {
            http: Client::new(),
            base_url: base_url.into(),
        }
    }

    pub async fn list_sessions(&self) -> Result<Vec<SessionInfo>> {
        let url = format!("{}/session", self.base_url);

        let sessions = self
            .http
            .get(url)
            .send()
            .await?
            .error_for_status()?
            .json::<Vec<SessionInfo>>()
            .await?;

        Ok(sessions)
    }

    pub async fn submit_prompt(&self, session_id: &str, input: PromptRequest) -> Result<()> {
        let url = format!("{}/session/{}/prompt", self.base_url, session_id);

        self.http
            .post(url)
            .json(&input)
            .send()
            .await?
            .error_for_status()?;

        Ok(())
    }

    pub async fn execute_tui_command(&self, command: &str) -> Result<()> {
        let url = format!("{}/tui/execute-command", self.base_url);

        self.http
            .post(url)
            .json(&serde_json::json!({ "command": command }))
            .send()
            .await?
            .error_for_status()?;

        Ok(())
    }
}

#[derive(Debug, Clone, Deserialize)]
pub struct SessionInfo {
    pub id: String,
    pub title: Option<String>,
    pub parent_id: Option<String>,
}

#[derive(Debug, Clone, Serialize)]
pub struct PromptRequest {
    pub text: String,
}
```

# Backend Event Stream

## `src/backend/events.rs`

```rust
use color_eyre::Result;
use eventsource_stream::Eventsource;
use futures_util::StreamExt;
use tokio::sync::mpsc;

use crate::{
    action::{Action, BackendEvent},
    backend::client::BackendClient,
};

pub fn spawn_event_stream(client: BackendClient, tx: mpsc::UnboundedSender<Action>) {
    tokio::spawn(async move {
        if let Err(error) = run_event_stream(client, tx.clone()).await {
            let _ = tx.send(Action::Toast(crate::action::Toast {
                level: crate::action::ToastLevel::Error,
                message: format!("Event stream disconnected: {error}"),
            }));
        }
    });
}

async fn run_event_stream(
    client: BackendClient,
    tx: mpsc::UnboundedSender<Action>,
) -> Result<()> {
    let url = format!("{}/event", client.base_url());

    let response = reqwest::get(url).await?;
    let mut stream = response.bytes_stream().eventsource();

    while let Some(event) = stream.next().await {
        let event = event?;

        if let Some(action) = map_sse_event(&event.event, &event.data) {
            let _ = tx.send(Action::BackendEvent(action));
        }
    }

    Ok(())
}

fn map_sse_event(name: &str, data: &str) -> Option<BackendEvent> {
    match name {
        "session.updated" => {
            let value: serde_json::Value = serde_json::from_str(data).ok()?;
            Some(BackendEvent::SessionUpdated {
                session_id: value.get("sessionID")?.as_str()?.to_string(),
            })
        }

        "permission.created" => {
            let value: serde_json::Value = serde_json::from_str(data).ok()?;
            Some(BackendEvent::PermissionCreated {
                request_id: value.get("requestID")?.as_str()?.to_string(),
            })
        }

        "question.created" => {
            let value: serde_json::Value = serde_json::from_str(data).ok()?;
            Some(BackendEvent::QuestionCreated {
                request_id: value.get("requestID")?.as_str()?.to_string(),
            })
        }

        "mcp.status" => Some(BackendEvent::McpStatus),
        "lsp.status" => Some(BackendEvent::LspStatus),

        _ => None,
    }
}
```

# Buffer-Level Signature Effects

To create a unique look that nobody confuses with OpenCode, use small buffer-level visual details.

Examples:

1. A subtle one-cell “activity rail” on the far left.
2. Ambient grid/pulse in the home screen.
3. Context-meter micro-gauges.
4. Tool execution shimmer.
5. Custom selected-row gutters.
6. Diff hunk heat markers.

Ratatui’s buffer model exposes cells with symbol and style data, and custom widgets can write directly to the buffer. That makes these signature touches possible without fighting a web-style component tree. ([Ratatui][7])

## `src/ui/widgets/ambient.rs`

```rust
use ratatui::prelude::*;

use crate::theme::Palette;

pub struct AmbientGrid {
    pub tick: u64,
    pub palette: Palette,
}

impl Widget for AmbientGrid {
    fn render(self, area: Rect, buf: &mut Buffer) {
        for y in area.y..area.y + area.height {
            for x in area.x..area.x + area.width {
                let phase = ((x as u64 * 7 + y as u64 * 13 + self.tick) % 29) as u8;

                if phase == 0 {
                    let cell = buf.cell_mut((x, y)).expect("cell exists");
                    cell.set_symbol("·");
                    cell.set_fg(self.palette.border_subtle);
                    cell.set_bg(self.palette.background);
                }
            }
        }
    }
}
```

This kind of subtle custom rendering is exactly how to create a distinct “world-class coding cockpit” identity.

# Final Ratatui Direction

The best version of this product is not “OpenCode with different colors.” It is a **Rust-native Ratatui coding cockpit**:

```text
APEX / workspace                         DARK ^⇧L      claude-sonnet
────────────────────────────────────────────────────────────────────
LEFT RAIL        │  Coding canvas
agent            │
model            │  YOU
context gauge    │  ...
files            │
permissions      │  ASSISTANT
MCP/LSP          │  ...
live tools       │
                 │  TOOL · read_file
                 │  src/main.rs
────────────────────────────────────────────────────────────────────
ASK  agent · model · context
╭──────────────────────────────────────────────────────────────────╮
│ Type prompt, /command, @file, !shell                             │
╰──────────────────────────────────────────────────────────────────╯
```

My updated recommendation is:

1. **Use Ratatui as the end-state frontend.**
2. **Keep the existing backend API.**
3. **Build the new UI around a left rail, central coding canvas, and command composer.**
4. **Use Ratatui custom widgets for the surfaces that define product identity.**
5. **Use `TestBackend` snapshot tests so the app stays beautiful at every terminal size.**
6. **Make dark/light mode a first-class shortcut with `Ctrl+Shift+L` plus fallback.**

This is the path that best satisfies your goals: Rust-native, visually distinct from OpenCode, technically strong, premium-feeling, and flexible enough to become a truly exceptional coding TUI.
