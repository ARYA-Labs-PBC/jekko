Absolutely — if you want to leverage Ratatui, I would revise the strategy from “restyle the existing OpenTUI/Solid TSX interface” into a **two-track plan**:

1. **Immediate visual break from OpenCode** inside the current TUI codebase.
2. **Strategic native Rust TUI rewrite using Ratatui** as the premium long-term direction.

The current archive’s TUI is not Rust; it is TypeScript/TSX using OpenTUI and SolidJS. Ratatui cannot be dropped into those TSX files directly. But the existing backend/API design is actually a very good fit for a native Rust frontend, because the TUI already communicates through backend events, session state, commands, and HTTP-style routes.

Ratatui is worth using if the goal is “world’s greatest coding app” in the terminal. Its official docs emphasize fast immediate-mode rendering, rich widgets such as charts, sparklines, tables, gauges, scrollable lists, progress bars, and responsive constraint-based layouts. That maps extremely well to the design I proposed: left rail, transcript canvas, command dock, agent telemetry, diff panels, and activity dashboards. ([Ratatui][1])

## Ratatui-first recommendation

My stronger recommendation is now:

```txt
Short term:
  Keep current OpenTUI frontend, but radically restyle it.

Long term:
  Build a new native Rust Ratatui frontend that talks to the existing backend.

Do not:
  Attempt to “mix” Ratatui widgets into the current TSX renderer.
```

The best architecture is a new binary:

# Ratatui-native design upgrade

Ratatui gives you a much more “engineered” terminal UI vocabulary than the current TSX view layer. The current UI is mostly boxes, text, scrollboxes, and manually styled borders. Ratatui lets you build a native immediate-mode terminal app with strong layout composition, reusable widgets, stateful widgets, and purpose-built visual components. Its widgets are designed to be combined and nested into complex TUIs, and the docs now describe the main crate as re-exporting the pieces app authors need, including widgets, backends, and layout systems. ([Docs.rs][2])

## 3. `crates/apex-tui/src/app.rs`

Core loop.

```rust
use color_eyre::Result;
use crossterm::event::{self, Event};
use ratatui::{backend::CrosstermBackend, Terminal};
use std::{io, time::Duration};

use crate::{
    backend::client::BackendClient,
    input::{actions::Action, keymap::map_key_event},
    state::AppState,
    ui,
};

pub struct App {
    client: BackendClient,
    state: AppState,
    should_quit: bool,
}

impl App {
    pub fn new(client: BackendClient) -> Self {
        Self {
            client,
            state: AppState::default(),
            should_quit: false,
        }
    }

    pub async fn run(&mut self) -> Result<()> {
        ratatui::init();

        let result = self.run_inner().await;

        ratatui::restore();

        result
    }

    async fn run_inner(&mut self) -> Result<()> {
        let backend = CrosstermBackend::new(io::stdout());
        let mut terminal = Terminal::new(backend)?;

        while !self.should_quit {
            terminal.draw(|frame| {
                ui::frame::render(frame, &mut self.state);
            })?;

            if event::poll(Duration::from_millis(16))? {
                if let Event::Key(key) = event::read()? {
                    let action = map_key_event(key, &self.state);
                    self.apply_action(action).await?;
                }
            }

            self.tick().await?;
        }

        Ok(())
    }

    async fn tick(&mut self) -> Result<()> {
        // Pull backend updates, process pending events, update animations.
        // The first implementation can poll. Later, switch to SSE/websocket stream.
        Ok(())
    }

    async fn apply_action(&mut self, action: Action) -> Result<()> {
        match action {
            Action::Quit => self.should_quit = true,

            Action::ToggleLeftRail => {
                self.state.layout.left_rail_open = !self.state.layout.left_rail_open;
            }

            Action::SubmitPrompt => {
                let text = self.state.prompt.input.clone();
                if !text.trim().is_empty() {
                    self.client.submit_prompt(&text).await?;
                    self.state.prompt.input.clear();
                }
            }

            Action::InputChar(ch) => {
                self.state.prompt.input.push(ch);
            }

            Action::Backspace => {
                self.state.prompt.input.pop();
            }

            Action::OpenCommandPalette => {
                self.state.overlay = Some(crate::state::Overlay::CommandPalette);
            }

            Action::Escape => {
                self.state.overlay = None;
            }

            Action::Noop => {}
        }

        Ok(())
    }
}
```

## 5. `crates/apex-tui/src/ui/layout.rs`

The Ratatui layout manager.

```rust
use ratatui::{
    layout::{Constraint, Direction, Layout, Rect},
};

#[derive(Debug, Clone, Copy)]
pub struct AppLayout {
    pub header: Rect,
    pub body: Rect,
    pub left_rail: Option<Rect>,
    pub transcript: Rect,
    pub prompt: Rect,
}

pub fn compute(area: Rect, left_rail_open: bool) -> AppLayout {
    let vertical = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(1),
            Constraint::Min(8),
        ])
        .split(area);

    let header = vertical[0];
    let body = vertical[1];

    let wide = area.width >= 118;
    let show_left = left_rail_open || wide;

    let horizontal = if show_left {
        Layout::default()
            .direction(Direction::Horizontal)
            .constraints([
                Constraint::Length(42),
                Constraint::Min(40),
            ])
            .split(body)
    } else {
        Layout::default()
            .direction(Direction::Horizontal)
            .constraints([Constraint::Min(40)])
            .split(body)
    };

    let left_rail = if show_left { Some(horizontal[0]) } else { None };
    let main = if show_left { horizontal[1] } else { horizontal[0] };

    let main_chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Min(8),
            Constraint::Length(7),
        ])
        .split(main);

    AppLayout {
        header,
        body,
        left_rail,
        transcript: main_chunks[0],
        prompt: main_chunks[1],
    }
}
```

This is much cleaner than the current TSX sidebar width and visibility logic. Ratatui’s layout system is designed for exactly this kind of adaptive terminal composition. ([Ratatui][3])

## 10. `crates/apex-tui/src/ui/command_palette.rs`

Ratatui can make the command palette feel like a first-class IDE feature.

```rust
use ratatui::{
    Frame,
    layout::{Alignment, Constraint, Direction, Layout, Rect},
    style::{Modifier, Style},
    text::{Line, Span},
    widgets::{Block, Borders, Clear, List, ListItem, Paragraph},
};

use crate::state::AppState;

use super::theme::Theme;

pub fn render(frame: &mut Frame, area: Rect, state: &AppState, theme: &Theme) {
    let popup = centered_rect(68, 70, area);

    frame.render_widget(Clear, popup);

    let block = Block::default()
        .borders(Borders::ALL)
        .border_style(Style::default().fg(theme.border_active))
        .style(Style::default().bg(theme.menu));

    frame.render_widget(block, popup);

    let inner = Rect {
        x: popup.x + 2,
        y: popup.y + 1,
        width: popup.width.saturating_sub(4),
        height: popup.height.saturating_sub(2),
    };

    let rows = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Length(2),
            Constraint::Min(3),
        ])
        .split(inner);

    frame.render_widget(
        Paragraph::new(Line::from(vec![
            Span::styled("◆ ", Style::default().fg(theme.primary).add_modifier(Modifier::BOLD)),
            Span::styled("Command", Style::default().fg(theme.text).add_modifier(Modifier::BOLD)),
            Span::styled("  type to filter", Style::default().fg(theme.muted)),
        ]))
        .alignment(Alignment::Left),
        rows[0],
    );

    let items: Vec<ListItem> = state
        .commands
        .iter()
        .enumerate()
        .map(|(idx, command)| {
            let selected = idx == state.command_palette.selected;
            let marker = if selected { "▸ " } else { "  " };
            let style = if selected {
                Style::default().fg(theme.text).bg(theme.element).add_modifier(Modifier::BOLD)
            } else {
                Style::default().fg(theme.muted).bg(theme.menu)
            };

            ListItem::new(Line::from(vec![
                Span::styled(marker, Style::default().fg(theme.primary).bg(theme.menu)),
                Span::styled(&command.title, style),
                Span::styled(
                    format!("  {}", command.category),
                    Style::default().fg(theme.muted).bg(if selected { theme.element } else { theme.menu }),
                ),
            ]))
        })
        .collect();

    frame.render_widget(
        List::new(items)
            .style(Style::default().bg(theme.menu))
            .highlight_style(theme.selected_style()),
        rows[1],
    );
}

fn centered_rect(percent_x: u16, percent_y: u16, area: Rect) -> Rect {
    let vertical = Layout::default()
        .direction(Direction::Vertical)
        .constraints([
            Constraint::Percentage((100 - percent_y) / 2),
            Constraint::Percentage(percent_y),
            Constraint::Percentage((100 - percent_y) / 2),
        ])
        .split(area);

    let horizontal = Layout::default()
        .direction(Direction::Horizontal)
        .constraints([
            Constraint::Percentage((100 - percent_x) / 2),
            Constraint::Percentage(percent_x),
            Constraint::Percentage((100 - percent_x) / 2),
        ])
        .split(vertical[1]);

    horizontal[1]
}
```

```rust
use color_eyre::Result;
use reqwest::Client;
use serde::{Deserialize, Serialize};

#[derive(Clone)]
pub struct BackendClient {
    base_url: String,
    http: Client,
}

impl BackendClient {
    pub fn new(base_url: String) -> Self {
        Self {
            base_url,
            http: Client::new(),
        }
    }

    pub async fn append_prompt(&self, text: &str) -> Result<bool> {
        let url = format!("{}/tui/append-prompt", self.base_url);

        let response = self
            .http
            .post(url)
            .json(&AppendPromptRequest {
                text: text.to_string(),
            })
            .send()
            .await?
            .json::<bool>()
            .await?;

        Ok(response)
    }

    pub async fn submit_prompt(&self, text: &str) -> Result<bool> {
        self.append_prompt(text).await?;

        let url = format!("{}/tui/submit-prompt", self.base_url);

        let response = self
            .http
            .post(url)
            .send()
            .await?
            .json::<bool>()
            .await?;

        Ok(response)
    }

    pub async fn execute_command(&self, command: &str) -> Result<bool> {
        let url = format!("{}/tui/execute-command", self.base_url);

        let response = self
            .http
            .post(url)
            .json(&ExecuteCommandRequest {
                command: command.to_string(),
            })
            .send()
            .await?
            .json::<bool>()
            .await?;

        Ok(response)
    }

    pub async fn select_session(&self, session_id: &str) -> Result<bool> {
        let url = format!("{}/tui/select-session", self.base_url);

        let response = self
            .http
            .post(url)
            .json(&SelectSessionRequest {
                session_id: session_id.to_string(),
            })
            .send()
            .await?
            .json::<bool>()
            .await?;

        Ok(response)
    }
}

#[derive(Debug, Serialize)]
struct AppendPromptRequest {
    text: String,
}

#[derive(Debug, Serialize)]
struct ExecuteCommandRequest {
    command: String,
}

#[derive(Debug, Serialize)]
#[serde(rename_all = "camelCase")]
struct SelectSessionRequest {
    session_id: String,
}
```

This gives the Rust TUI a clean starting point. Later, add typed models generated from OpenAPI if desired.

pub fn map_key_event(key: KeyEvent, _state: &AppState) -> Action {
    match (key.code, key.modifiers) {
        (KeyCode::Char('q'), KeyModifiers::CONTROL) => Action::Quit,
        (KeyCode::Char('p'), KeyModifiers::CONTROL) => Action::OpenCommandPalette,

# What Ratatui unlocks visually

## 1. Left rail telemetry

Use:

```txt
Gauge      context budget
Sparkline  agent activity
List       modified files
Table      provider/model/tool state
Tabs       workspace/session/tool panes
Scrollbar  transcript position
Chart      token/cost/activity trends
Canvas     optional tiny topology map
```

Ratatui officially provides widget examples for these categories, including Gauge, List, Paragraph, Scrollbar, Sparkline, Table, Tabs, Chart, Canvas, and BarChart. ([Ratatui][4])

## 2. Premium diff view

Use `similar` to compute line changes, then render a custom Ratatui widget:

```txt
◇ src/provider/auth.ts
  @@ authenticateProvider @@

  - old token validation
  + new token validation
  + refreshed provider cache
```

Use:

```rust
Span::styled("+", Style::default().fg(theme.success))
Span::styled("-", Style::default().fg(theme.error))
```

## 3. Permission modal as native action sheet

```txt
╭─ Permission Required ─────────────────────────╮
│ tool     edit_file                            │
│ target   src/provider/auth.ts                 │
│                                                │
│ ▸ Allow once    Always allow    Deny           │
╰───────────────────────────────────────────────╯
```

Ratatui’s `Clear` widget is useful for modal overlays, and blocks/borders make this straightforward. The official widget docs describe `Clear` as clearing the area it occupies, useful when rendering over previously drawn widgets. ([Docs.rs][2])

## 4. Better performance and reliability

Ratatui’s website describes immediate-mode rendering, sub-millisecond rendering, and pure Rust reliability as core benefits. For a terminal app with streaming output, telemetry, scrollback, syntax-heavy text, and many dynamic widgets, that is exactly the direction I’d want. ([Ratatui][1])

# Bottom-line recommendation

Yes: leverage Ratatui, but do it boldly.

The best final product is not “OpenTUI with a Ratatui-inspired theme.” It is a **native Rust Ratatui coding workbench** that uses the existing backend as its brain.

That gives you:

```txt
Native Rust performance
True terminal-first layout control
Left rail dashboard
Professional widgets
Cleaner state/render separation
A completely different identity from OpenCode
A much stronger foundation for a world-class coding TUI
```

I would still apply the immediate TSX redesign for speed, but the flagship experience should move to Ratatui.
