Yes — that is the stronger long-term direction.

The current uploaded code is an OpenTUI/Solid TypeScript renderer, so Ratatui cannot be “dropped into” those TSX files directly. To truly leverage Ratatui, I would split the plan into two tracks:

1. **Immediate track:** apply the visual redesign inside the existing OpenTUI renderer so the product quickly stops looking like OpenCode.
2. **Strategic track:** build a new **Rust/Ratatui-native renderer** that consumes the same backend API, SSE events, sessions, prompts, and command endpoints.

The backend architecture you already have is good enough for this. The frontend renderer is the part that should evolve.

import path from "node:path"

export async function launchRatatuiTui(input: {
  url: string
  directory: string
  sessionID?: string
  continue?: boolean
  fork?: boolean
  model?: string
  agent?: string
  prompt?: string
  headers?: Record<string, string>

  const child = spawn(bin, [], {
    cwd: input.directory,
    stdio: ["inherit", "inherit", "inherit"],
    env: {
      ...process.env,

      JEKKO_TUI_DIRECTORY: input.directory,
      JEKKO_TUI_SESSION_ID: input.sessionID ?? "",
      JEKKO_TUI_CONTINUE: input.continue ? "1" : "0",
      JEKKO_TUI_FORK: input.fork ? "1" : "0",
      JEKKO_TUI_MODEL: input.model ?? "",
      JEKKO_TUI_AGENT: input.agent ?? "",
      JEKKO_TUI_PROMPT: input.prompt ?? "",
      JEKKO_TUI_HEADERS: JSON.stringify(input.headers ?? {}),
    },
  })

    child.on("exit", (code) => {
      if (code === 0 || code === null) resolve()
      else reject(new Error(`Ratatui renderer exited with code ${code}`))

    child.on("error", reject)

## App Event Loop

```rust
// crates/jekko-tui/src/app.rs

use std::time::{Duration, Instant};

use color_eyre::Result;
use crossterm::event::{self, Event, KeyCode, KeyModifiers};
use ratatui::DefaultTerminal;

use crate::{
    backend::BackendClient,
    config::Config,
    input::{Action, Keymap},
    state::AppState,
    ui,
};

pub struct App {
    state: AppState,
    backend: BackendClient,
    keymap: Keymap,
    running: bool,
    last_tick: Instant,
}

impl App {
    pub async fn new(config: Config) -> Result<Self> {
        let backend = BackendClient::new(config.backend.clone())?;
        let state = AppState::bootstrap(config).await?;

        Ok(Self {
            state,
            backend,
            keymap: Keymap::default(),
            running: true,
            last_tick: Instant::now(),
        })
    }

    pub async fn run(&mut self, terminal: &mut DefaultTerminal) -> Result<()> {
        self.backend.start_event_stream().await?;

        while self.running {
            terminal.draw(|frame| ui::render(frame, &mut self.state))?;

            self.drain_backend_events().await?;
            self.handle_terminal_events().await?;
            self.tick().await?;
        }

        Ok(())
    }

    async fn handle_terminal_events(&mut self) -> Result<()> {
        while event::poll(Duration::from_millis(8))? {
            match event::read()? {
                Event::Key(key) => {
                    if key.code == KeyCode::Char('c') && key.modifiers.contains(KeyModifiers::CONTROL) {
                        self.running = false;
                        return Ok(());
                    }

                    if let Some(action) = self.keymap.resolve(key) {
                        self.dispatch(action).await?;
                    }
                }
                Event::Resize(_, _) => {
                    self.state.layout_dirty = true;
                }
                _ => {}
            }
        }

        Ok(())
    }

    async fn dispatch(&mut self, action: Action) -> Result<()> {
        match action {
            Action::Quit => self.running = false,
            Action::ToggleThemeMode => self.state.theme.toggle_mode(),
            Action::SubmitPrompt => {
                let text = self.state.prompt.take();
                self.backend.append_prompt(&text).await?;
                self.backend.submit_prompt().await?;
            }
            Action::OpenCommandPalette => {
                self.state.modal.open_command_palette();
            }
            Action::OpenThemes => {
                self.backend.execute_command("theme.switch").await?;
            }
            Action::ScrollUp => self.state.timeline.scroll_up(),
            Action::ScrollDown => self.state.timeline.scroll_down(),
            Action::ToggleLeftRail => self.state.left_rail.toggle(),
        }

        Ok(())
    }

    async fn drain_backend_events(&mut self) -> Result<()> {
        while let Some(evt) = self.backend.try_next_event()? {
            self.state.apply_backend_event(evt);
        }
        Ok(())
    }

    async fn tick(&mut self) -> Result<()> {
        if self.last_tick.elapsed() >= Duration::from_millis(100) {
            self.state.animation.tick();
            self.last_tick = Instant::now();
        }

        Ok(())
    }
}
```

## Nova Layout in Ratatui

This is where Ratatui really shines. Build the whole product around a fixed left rail, flexible center, top ribbon, and bottom command capsule.

```rust
// crates/jekko-tui/src/ui/layout.rs

use ratatui::layout::{Constraint, Layout, Rect};

pub const RAIL_WIDTH: u16 = 44;
pub const RAIL_BREAKPOINT: u16 = 118;
pub const RIBBON_HEIGHT: u16 = 2;
pub const PROMPT_HEIGHT: u16 = 7;

#[derive(Debug, Clone, Copy)]
pub struct Regions {
    pub ribbon: Rect,
    pub rail: Option<Rect>,
    pub runway: Rect,
    pub prompt: Rect,
    pub overlay: Rect,
}

pub fn regions(area: Rect, rail_visible: bool) -> Regions {
    let [ribbon, body, prompt] = Layout::vertical([
        Constraint::Length(RIBBON_HEIGHT),
        Constraint::Min(0),
        Constraint::Length(PROMPT_HEIGHT),
    ])
    .areas(area);

    let wide = area.width >= RAIL_BREAKPOINT;

    let (rail, runway) = if rail_visible && wide {
        let [rail, runway] = Layout::horizontal([
            Constraint::Length(RAIL_WIDTH),
            Constraint::Min(40),
        ])
        .areas(body);

        (Some(rail), runway)
    } else {
        (None, body)
    };

    Regions {
        ribbon,
        rail,
        runway,
        prompt,
        overlay: body,
    }
}
```

This gives you the exact product direction: permanent left intelligence rail on wide terminals; overlay left rail on small terminals.

## Message Cards as Custom Widgets

Ratatui explicitly supports custom widgets through the `Widget` trait, where a custom struct renders into a `Buffer`. ([Ratatui][5])

This is how you make the app feel proprietary and premium.

```rust
// crates/jekko-tui/src/ui/widgets/message.rs

use ratatui::{
    buffer::Buffer,
    layout::{Margin, Rect},
    style::{Modifier, Style},
    text::{Line, Text},
    widgets::{Block, Borders, Paragraph, Widget, Wrap},
};

use crate::theme::NovaTheme;

#[derive(Debug, Clone, Copy)]
pub enum MessageTone {
    User,
    Assistant,
    Tool,
    Thinking,
    Error,
}

pub struct MessageCard<'a> {
    pub theme: NovaTheme,
    pub tone: MessageTone,
    pub title: Line<'a>,
    pub body: Text<'a>,
}

impl Widget for MessageCard<'_> {
    fn render(self, area: Rect, buf: &mut Buffer) {
        let border = match self.tone {
            MessageTone::User => self.theme.primary,
            MessageTone::Assistant => self.theme.border,
            MessageTone::Tool => self.theme.secondary,
            MessageTone::Thinking => self.theme.border_subtle,
            MessageTone::Error => self.theme.error,
        };

        let block = Block::default()
            .borders(Borders::TOP | Borders::BOTTOM)
            .border_style(Style::default().fg(border))
            .style(Style::default().bg(self.theme.panel).fg(self.theme.text));

        block.render(area, buf);

        let inner = area.inner(Margin {
            horizontal: 2,
            vertical: 1,
        });

        let title_area = Rect {
            x: inner.x,
            y: inner.y,
            width: inner.width,
            height: 1,
        };

        let body_area = Rect {
            x: inner.x,
            y: inner.y.saturating_add(2),
            width: inner.width,
            height: inner.height.saturating_sub(2),
        };

        Paragraph::new(self.title)
            .style(Style::default().bg(self.theme.panel).fg(border).add_modifier(Modifier::BOLD))
            .render(title_area, buf);

        Paragraph::new(self.body)
            .style(Style::default().bg(self.theme.panel).fg(self.theme.text))
            .wrap(Wrap { trim: false })
            .render(body_area, buf);
    }
}

            Span::styled("^T", Style::default().fg(t.accent).add_modifier(Modifier::BOLD)),
            Span::styled(" white/dark   ", Style::default().fg(t.muted)),
            Span::styled("^K", Style::default().fg(t.secondary).add_modifier(Modifier::BOLD)),
            Span::styled(" command", Style::default().fg(t.muted)),
        ]))
        .style(Style::default().bg(t.element))
        .render(footer, buf);
    }
}
```

## Canvas Hero for Home Screen

Ratatui’s `Canvas` widget draws arbitrary shapes using drawing characters, and examples show Braille markers and map/shape rendering. ([Ratatui][6])

Use this for a home screen that no one will confuse with OpenCode:

```text
✦ Jekko Nova
terminal-native engineering cockpit

     ⟡
   ╱   ╲       build plans, patches, shell, context
  ╱  ◇  ╲      safe autonomous coding
 ╱_______╲     one command surface
```

A `Canvas`-based animated “aurora” hero can use Braille or half-block markers behind the logo. Keep it subtle; terminals can get visually noisy fast.

Suggested file:

```text
crates/jekko-tui/src/ui/widgets/canvas_hero.rs
```

Concept:

```rust
use ratatui::{
    buffer::Buffer,
    layout::Rect,
    style::Color,
    symbols::Marker,
    widgets::{
        canvas::{Canvas, Line},
        Widget,
    },
};

use crate::theme::NovaTheme;

pub struct NovaHero {
    pub theme: NovaTheme,
    pub tick: u64,
}

impl Widget for NovaHero {
    fn render(self, area: Rect, buf: &mut Buffer) {
        let canvas = Canvas::default()
            .marker(Marker::Braille)
            .x_bounds([0.0, 100.0])
            .y_bounds([0.0, 100.0])
            .paint(|ctx| {
                for i in 0..6 {
                    let offset = ((self.tick + i * 7) % 40) as f64;
                    ctx.draw(&Line {
                        x1: 10.0,
                        y1: 20.0 + offset,
                        x2: 90.0,
                        y2: 80.0 - offset,
                        color: match i % 3 {
                            0 => self.theme.primary,
                            1 => self.theme.secondary,
                            _ => self.theme.accent,
                        },
                    });
                }
            });

        canvas.render(area, buf);
    }
}
```

## White/Dark Shortcut in Ratatui

In the Rust keymap:

```rust
// crates/jekko-tui/src/input/keymap.rs

use crossterm::event::{KeyCode, KeyEvent, KeyModifiers};

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Action {
    Quit,
    SubmitPrompt,
    ToggleThemeMode,
    OpenCommandPalette,
    OpenThemes,
    ToggleLeftRail,
    ScrollUp,
    ScrollDown,
}

#[derive(Debug, Default)]
pub struct Keymap;

impl Keymap {
    pub fn resolve(&self, key: KeyEvent) -> Option<Action> {
        match (key.code, key.modifiers) {
            (KeyCode::Char('c'), m) if m.contains(KeyModifiers::CONTROL) => Some(Action::Quit),
            (KeyCode::Char('t'), m) if m.contains(KeyModifiers::CONTROL) => Some(Action::ToggleThemeMode),
            (KeyCode::Char('k'), m) if m.contains(KeyModifiers::CONTROL) => Some(Action::OpenCommandPalette),
            (KeyCode::Char('b'), m) if m.contains(KeyModifiers::CONTROL) => Some(Action::ToggleLeftRail),
            (KeyCode::Enter, _) => Some(Action::SubmitPrompt),
            (KeyCode::PageUp, _) => Some(Action::ScrollUp),
            (KeyCode::PageDown, _) => Some(Action::ScrollDown),
            _ => None,
        }
    }
}
```

I would present this shortcut as:

```text
^T white/dark
```

Avoid `Ctrl+Shift+T` as the primary default because many terminals reserve or inconsistently transmit it. Use `Ctrl+T` by default, and optionally support `Ctrl+Shift+T` when the terminal reports it clearly.

## Best Final Direction

My strongest recommendation is:

**Ship the visual redesign in the current OpenTUI frontend only as a bridge. Build the actual world-class product as `crates/jekko-tui`, a Ratatui-native renderer.**

That gives you:

* a truly Rust-native TUI;
* a completely different look from OpenCode;
* a left-side intelligence rail by design;
* first-class white/dark mode;
* much better custom widgets;
* real dashboard components using gauges, sparklines, tables, lists, scrollbars, and canvas;
* clean separation between backend intelligence and frontend rendering;
* the ability to eventually remove the recognizable TSX/OpenTUI visual lineage.

Ratatui is the right foundation for the “world’s greatest coding TUI” version. The current backend can remain; the renderer should become native.
