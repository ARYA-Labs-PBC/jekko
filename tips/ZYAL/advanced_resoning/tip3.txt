I studied the public Jekko/ZYAL docs and examples closely, especially the mission, canonical spec, full-power runbook, memory system notes, and AutoResearch loop. I did **not** run a local build, so these recommendations are based on the repository materials I inspected.

## My strongest recommendation

Do **not** make a “chat swarm.” Make a **host-governed reasoning gauntlet**.

The most powerful ZYAL design for weaker models is one where the **host/runtime owns the loop**, weaker models are treated as bounded specialists, every claim is routed through structured evidence, and long-term memory is written only after verification. That matches ZYAL’s own philosophy: it describes itself as a host-enforced YAML operating contract where the host controls lifecycle, bounds, evidence, durability, and approval rather than trusting the model to remember rules. ([GitHub][1])

The punchline: weaker models become much more useful when they are not asked to “be smart” end-to-end. They should be asked to produce **small, checkable artifacts** inside a larger workflow that supplies decomposition, diversity, adversarial review, memory, verification, and rollback.

---

## Key ZYAL reading: what matters most

ZYAL’s existing direction is already excellent for this. The mission explicitly calls out multi-agent orchestration, supervisor-dispatched builder agents, isolated worktrees, critics on different providers, evidence-gated promotion, hypothesis tournaments, anti-vibe quality rules, cost control, and observability as core endgame capabilities. ([GitHub][1])

The important caveat is implementation status. The spec presents many blocks as part of the 2.6 contract, but the mission notes that several higher-power v2.1–v2.4 blocks are currently parser/preview contracts rather than fully enforced runtime behavior. So the design should use runtime-enforced pieces now, while treating advanced blocks such as models, experiments, quality, capabilities, budgets, research, taint, memory-kernel, and evidence-graph as the roadmap to harden. ([GitHub][2])

The existing “full power” example is very close to the desired pattern: workflow phases, multi-tier memory, typed evidence, approvals, governed skills, sandboxing, capability leases, anti-vibe checks, budgets, triggers, rollback, done conditions, repo graph control, and observability. ([GitHub][3])

# The design I would build: **ZYAL Reasoning Gauntlet**

The system should have nine phases:

1. **Intake**
2. **Memory/context refresh**
3. **Problem decomposition**
4. **Blind hypothesis tournament**
5. **Adversarial critique**
6. **Synthesis**
7. **External verification**
8. **Evidence-gated promotion**
9. **Memory writeback**

This is not just “many models vote.” Voting is weak. The powerful thing is **diverse independent attempts + adversarial pressure + evidence + durable memory**.

---

## 1. Use the host as the real “reasoning control plane”

For weak models, the orchestration layer must be smarter than any individual worker. The host should own:

* state transitions;
* task splitting;
* worker assignment;
* context packing;
* stop conditions;
* retry policy;
* evidence requirements;
* approval gates;
* memory writes;
* final answer assembly.

This is exactly aligned with ZYAL’s “host owns the loop” principle and its state-machine workflow model, where phases such as discover, plan, implement, verify, and promote are gated by evidence and approvals. ([GitHub][1])

The key design rule: **no model decides that the task is done.** A model may propose done; the host checks done.

ZYAL’s full-power example already expresses this well: terminal success requires workflow completion, evidence completion, merge review, rollback verification, and no critical objections, while explicitly forbidding model-only claims and “tests not run.” ([GitHub][3])

---

## 2. Split weak models into narrow roles

Weak models fail when asked to solve broad, ambiguous tasks. They perform much better when assigned tiny, typed roles.

Recommended roles:

| Role                      | Purpose                                       | Weak-model-friendly output      |
| ------------------------- | --------------------------------------------- | ------------------------------- |
| **Supervisor**            | Own task state and routing, not content truth | task spec, risk map, next phase |
| **Decomposer**            | Split hard task into claims/subproblems       | dependency graph                |
| **Context scout**         | Retrieve relevant memory/files/sources        | cited context pack              |
| **Hypothesis worker**     | Solve one slice independently                 | claim report                    |
| **Counterexample worker** | Try to break a claim                          | objection report                |
| **Critic**                | Review blind solutions                        | scored critique                 |
| **Verifier**              | Run tests/checkers/tools                      | receipt bundle                  |
| **Synthesizer**           | Merge surviving claims                        | final structured answer         |
| **Memory curator**        | Decide what persists                          | verified memory diff            |

ZYAL already supports the shape of this with agents, worker pools, isolation modes, fan-out/reduce, workflow, evidence, and memory blocks. The spec defines worker pools with counts, isolation options such as git worktrees or same-session execution, and integration settings. ([GitHub][2])

For the weakest models, make each role even narrower. For example, do not ask one model to “critique the solution.” Ask one model to find missing assumptions, another to find invalid inference steps, another to generate counterexamples, another to check source grounding.

---

## 3. Make tournaments the core reasoning primitive

The most important ZYAL pattern for stronger reasoning is the **hypothesis tournament**.

For every hard problem, run multiple isolated lanes:

* each lane gets the same task spec;
* each lane gets a different reasoning strategy;
* lanes cannot see each other’s outputs;
* each lane must emit structured claims, assumptions, evidence, and uncertainty;
* critics judge blind;
* reducer picks the best verified synthesis;
* failed lanes are preserved as negative memory.

This is directly supported by the ZYAL examples and mission. The examples list a hypothesis tournament using disjoint experiments, isolated worktrees, blind cross-provider judging, and negative-memory learning. ([GitHub][4]) The full-power runbook’s experiment section uses a disjoint tournament, distinct plans, multiple lanes, scoring weights, blind judging by a different-provider critic, best-verified-patch reduction, and failed-lane preservation as negative memory. ([GitHub][3])

For non-coding reasoning, “worktree” becomes “scratch context” or “claim graph branch.” The same tournament logic applies.

Best lane strategies:

* **Direct solution lane**
* **First-principles lane**
* **Counterexample-first lane**
* **Analogy/transfer lane**
* **Formalization lane**
* **Minimal-assumption lane**
* **Search/retrieval-heavy lane**
* **Skeptical/null-hypothesis lane**
* **Adversarial red-team lane**
* **Compression/simplicity lane**

The reducer should not choose the most confident lane. It should choose the lane whose claims survive the most checks.

---

## 4. Memory should be a verified reasoning substrate, not a transcript dump

Memory is where this can become dramatically more powerful over time. But memory is also where weak models can poison the system.

I would use five memory stores:

1. **Task memory**
   Ephemeral scratch for the current run: task spec, assumptions, partial decompositions, unresolved objections.

2. **Claim/evidence ledger**
   Append-only graph of claims, evidence, objections, verifier receipts, and decisions.

3. **Negative memory**
   Failed approaches, false assumptions, bad patches, invalid proofs, hallucinated sources, recurring traps.

4. **Concept/strategy memory**
   Reusable distilled methods: “For this class of task, this decomposition worked,” “This checker caught failures,” “This source type was unreliable.”

5. **Domain/repo memory**
   Facts about the codebase, project, domain, architecture, APIs, tests, constraints, and prior decisions.

ZYAL’s memory block already supports scoped stores, retention, compression, write/read policy, searchability, redaction, and provenance tracking. ([GitHub][2]) The full-power runbook uses task context, failed approaches, architectural decisions, redaction, and hash-chain provenance. ([GitHub][3])

The deeper SmartMemory/Cogcore design is especially relevant: it describes deterministic append-only memory, a hash-chained ledger, BM25 retrieval, Hebbian co-activation, FSRS-like strength/half-life, concept kernels, topic hardening, provenance, validity, privacy, source quality, utility, contradiction/supersession links, and deterministic recall mutation through WAL entries. ([GitHub][5])

My strongest memory rule:

> **Permanent memory may only be written by a memory curator after promotion, with provenance, verifier receipts, and contradiction links.**

Never let raw worker outputs directly enter long-term memory.

---

## 5. Collaboration should be structured blackboard, not free-form debate

Weak models talking freely can spiral into agreement, confusion, or role drift. The collaboration layer should be an append-only blackboard with typed records.

Every worker should write records like:

```yaml
claim:
  id: C-042
  parent: C-017
  proposition: "The bug is caused by stale cache invalidation."
  assumptions:
    - A-003
  evidence:
    - E-019
    - E-021
  objections:
    - O-006
  confidence:
    value: 0.61
    basis: "test reproduction + code path match"
  status: "contested"
```

The repo already uses an append-only coordination pattern for ZYAL work, with recent receipts and active claims living above the log, and a separate agent chat file acting as an inter-agent coordination channel. ([GitHub][6])

For maximum reasoning, the blackboard should enforce:

* every claim has an ID;
* every claim has assumptions;
* every claim has evidence or is marked speculative;
* every objection attaches to a claim;
* every synthesis cites claim IDs;
* every final answer cites evidence IDs;
* every unresolved critical objection blocks promotion.

This converts “conversation” into a reasoning graph.

---

## 6. Verification must dominate majority vote

A majority of weak models can be confidently wrong. Use voting only when verification is impossible.

Preferred hierarchy:

1. **Formal proof/checker**
2. **Executable test**
3. **Deterministic validation script**
4. **Source-grounded evidence**
5. **Independent adversarial review**
6. **Blind consensus**
7. **Model self-confidence**

ZYAL’s evidence block is already pointed in this direction: it requires typed proof bundles before promotion, with bundle format, signing, and archiving. ([GitHub][2]) The full-power runbook requires evidence such as tests, typecheck, security, blind critic, rollback, ownership, and regression proof before promotion. ([GitHub][3])

For non-code reasoning, create analogues:

* math: proof checker, symbolic algebra, counterexample search;
* planning: constraint solver, budget checker, timeline validator;
* research: source receipts, citation triangulation, contradiction search;
* strategy: premortem, scenario tests, assumptions ledger;
* writing: rubric scoring, factuality check, audience fit check;
* data analysis: reproducible notebook, schema validation, sensitivity test.

The final reducer should prefer one checked claim over five unchecked claims.

---

## 7. Model routing: make weak models strong through specialization and calibration

ZYAL already has a models concept with planner, builder, critic, summarizer routes, fallback chains, critic separation, and confidence caps. ([GitHub][1]) The full-power runbook similarly routes plan, build, critic, and summarizer profiles, and requires the critic to differ where possible. ([GitHub][3])

I would implement an adaptive model-power policy:

### When models are very weak

Use:

* more decomposition;
* more lanes;
* smaller outputs;
* strict schemas;
* more retrieval;
* more tool verification;
* more critic passes;
* lower trust in memory writes;
* no free-form final synthesis from one worker.

### When models are medium strength

Use:

* fewer but deeper lanes;
* more synthesis responsibility;
* critics can evaluate multiple claims;
* memory curator can propose broader abstractions.

### When a stronger model is available

Spend it on:

* decomposition review;
* final synthesis;
* adversarial critique;
* ambiguity resolution;
* high-risk approval recommendations.

Do **not** waste the strongest model on bulk retrieval, formatting, simple test interpretation, or generating many ordinary variants.

The routing score should be empirical, not assumed. Track each model’s:

* lane win rate;
* verifier pass rate;
* hallucination rate;
* objection quality;
* memory usefulness;
* cost per promoted claim;
* failure modes by task type.

Then route by observed performance.

---

## 8. Use negative memory aggressively

This is one of the highest-leverage ideas in the repo.

ZYAL’s tournament examples and full-power runbook preserve failed lanes as negative memory. ([GitHub][4]) The AutoResearch design also has a negative-memory schema and an append-only ledger for failed approaches, with deduplication before feeding prior failures back into future prompts. ([GitHub][7])

Negative memory should store:

* failed hypothesis;
* why it seemed plausible;
* exact disproof;
* verifier command or evidence;
* model/lane that proposed it;
* task/domain tags;
* what to try instead.

This is how a weak cluster becomes better over time: not just by remembering answers, but by remembering **traps**.

---

## 9. Add shadow evaluation to prevent benchmark gaming

The AutoResearch loop has a strong pattern here: workers are isolated in worktrees, cannot edit trusted core/scoring/reference surfaces, and promotions require clean gates, determinism, no regressions, reference drift checks, and shadow checks. ([GitHub][7]) It also describes a shadow suite used to detect overfitting and pause promotion when public-vs-shadow gaps become suspicious. ([GitHub][7])

For reasoning tasks, create both:

* **public checks**, visible to workers;
* **shadow checks**, hidden from workers and used by the reducer.

Examples:

* hidden counterexamples;
* withheld source snippets;
* private regression tasks;
* alternate rubric judge;
* unseen unit tests;
* contradiction probes.

This prevents the cluster from learning to satisfy the visible rubric while failing the real objective.

# Concrete ZYAL runbook skeleton

This is not meant as a drop-in guaranteed-valid file; it is the design shape I would implement or compile into currently enforced ZYAL primitives. It intentionally leans on workflow, fan-out, evidence, memory, guardrails, assertions, budgets, approvals, and observability, while treating preview-only advanced blocks as contracts to harden.

```yaml
version: "2.6.0"
intent: daemon
confirm: RUN_FOREVER
id: max-reasoning-weak-cluster

job:
  name: "ZYAL Reasoning Gauntlet"
  objective: >
    Maximize reasoning quality from weaker models by using host-owned
    decomposition, blind diverse hypothesis lanes, adversarial critique,
    evidence verification, synthesis, and governed memory writeback.
  risk:
    - weak_model_hallucination
    - majority_wrong_consensus
    - memory_poisoning
    - unverified_final_claims

loop:
  mode: forever
  max_steps_per_task: 64
  stop_on:
    - human_pause
    - unresolved_critical_objection
    - evidence_gate_failed
    - budget_exceeded

context:
  mode: hybrid
  compact_every: 8
  hard_clear_every: 24
  preserve:
    - task_spec
    - success_criteria
    - claim_graph
    - evidence_index
    - unresolved_objections
    - negative_memory_hits
    - final_decision

agents:
  supervisor:
    role: control_plane
    allowed_actions: [route, summarize_state, request_evidence]
  decomposer:
    role: split_problem
    output_schema: decomposition_report
  scout:
    role: retrieve_context
    output_schema: context_pack
  solver:
    role: independent_hypothesis
    output_schema: claim_report
  critic:
    role: adversarial_review
    output_schema: objection_report
  verifier:
    role: external_check
    output_schema: evidence_receipt
  synthesizer:
    role: merge_survivors
    output_schema: synthesis_report
  memory_curator:
    role: post_promotion_memory
    output_schema: memory_diff

fleet:
  max_workers: 20
  isolation: hybrid
  policy:
    prefer_parallel_lanes_for_low_power_models: true
    prevent_lane_cross_talk: true
    require_worker_receipts: true

workflow:
  type: state_machine
  initial: intake
  states:
    intake:
      agent: supervisor
      produces:
        - task_spec
        - success_criteria
        - risk_map
      next: memory_refresh

    memory_refresh:
      agent: scout
      reads:
        - domain_memory
        - concept_memory
        - negative_memory
      produces:
        - context_pack
        - negative_memory_hits
      next: decompose

    decompose:
      fan_out:
        agent: decomposer
        lanes: 3
        diversity:
          - first_principles
          - dependency_graph
          - failure_modes
      reduce: supervisor
      produces:
        - decomposition_graph
      next: hypothesis_tournament

    hypothesis_tournament:
      fan_out:
        agent: solver
        lanes: adaptive
        min_lanes: 5
        max_lanes: 20
        isolation: scratch_branch
        diversity:
          - direct_solution
          - counterexample_first
          - formalization
          - retrieval_heavy
          - minimal_assumption
          - adversarial_null
      produces:
        - lane_claim_reports
      next: adversarial_critique

    adversarial_critique:
      fan_out:
        agent: critic
        blind: true
        require_different_model_family_when_available: true
        targets:
          - assumptions
          - invalid_inference
          - missing_evidence
          - contradictions
          - edge_cases
      produces:
        - objection_reports
      next: synthesis

    synthesis:
      agent: synthesizer
      inputs:
        - lane_claim_reports
        - objection_reports
        - evidence_index
      rules:
        - reject_model_only_claims
        - cite_claim_ids
        - cite_evidence_ids
        - preserve_uncertainty
      produces:
        - candidate_answer
        - surviving_claim_graph
      next: verify

    verify:
      agent: verifier
      checks:
        - executable_or_formal_checks_when_available
        - source_grounding
        - contradiction_search
        - regression_or_shadow_eval
        - unresolved_objection_scan
      produces:
        - verification_receipts
      next:
        pass: promote
        fail: repair_or_escalate

    repair_or_escalate:
      agent: supervisor
      policy:
        if_failure_is_local: spawn_targeted_lanes
        if_failure_is_conceptual: return_to_decompose
        if_conflict_persists: request_human_approval
      next: hypothesis_tournament

    promote:
      agent: supervisor
      requires:
        - evidence_complete
        - no_unresolved_critical_objections
        - verifier_passed
        - budget_ok
      produces:
        - final_answer
        - promotion_receipt
      next: memory_writeback

    memory_writeback:
      agent: memory_curator
      writes:
        - verified_claims_to_domain_memory
        - failed_approaches_to_negative_memory
        - reusable_methods_to_concept_memory
      rules:
        - append_only_by_default
        - provenance_required
        - contradiction_links_required
        - no_raw_worker_transcripts
      next: done

memory:
  stores:
    task_context:
      scope: task
      retention: until_done
      searchable: true
      write_policy: any_agent_structured

    claim_ledger:
      scope: task
      retention: permanent_after_promotion
      searchable: true
      write_policy: host_only
      provenance: hash_chain

    negative_memory:
      scope: global
      retention: permanent
      searchable: true
      write_policy: memory_curator_after_verification
      compression: dedupe_by_failure_signature

    concept_memory:
      scope: global
      retention: permanent
      searchable: true
      write_policy: memory_curator_after_promotion
      compression: distill_reusable_strategy

    domain_memory:
      scope: repo_or_domain
      retention: permanent
      searchable: true
      write_policy: approval_or_verified_receipt

evidence:
  require_before_promote:
    - claim_graph_complete
    - verifier_receipts_present
    - critic_objections_resolved_or_acknowledged
    - source_grounding_for_external_facts
    - shadow_check_when_available
  archive:
    format: receipt_bundle
    hash: sha256

quality:
  fail_closed: true
  anti_vibe:
    - no_model_only_claims
    - no_fake_evidence
    - no_unresolved_critical_objections
    - no_confidence_without_basis
    - no_memory_write_without_promotion

assertions:
  structured_outputs_required: true
  claim_report_schema:
    required:
      - claim_id
      - proposition
      - assumptions
      - evidence_refs
      - uncertainty
      - possible_counterexamples
      - verification_suggestions

budgets:
  per_task:
    max_parallel_workers: 20
    max_repair_cycles: 4
    max_tokens_soft: adaptive
    max_cost_soft: adaptive
  escalation:
    on_budget_pressure: reduce_lanes_then_request_approval

approvals:
  require_human_for:
    - persistent_memory_overwrite
    - high_risk_final_answer
    - unresolved_critical_objection
    - capability_expansion
    - external_side_effects

observability:
  metrics:
    - lane_diversity
    - contradiction_rate
    - verifier_pass_rate
    - critic_hit_rate
    - memory_hit_rate
    - negative_memory_reuse
    - cost_per_promoted_claim
    - human_override_rate
    - stale_memory_detected
```

# Support I would add to Jekko/ZYAL

## A. `zyal-reason` reducer tool

Add a first-class reducer that understands claim graphs.

It should:

* ingest lane reports;
* deduplicate claims;
* identify contradictions;
* cluster equivalent arguments;
* attach objections;
* demand evidence;
* select surviving claims;
* produce a final synthesis only from verified or explicitly caveated claims.

This becomes the core “reasoning compiler” between weak workers and final output.

## B. Claim/evidence graph

Add a persistent graph schema:

```yaml
Claim:
  id
  proposition
  assumptions
  dependencies
  evidence_refs
  objection_refs
  status: proposed | supported | contradicted | verified | rejected
  confidence:
    numeric
    basis
    calibrated_by

Evidence:
  id
  type: test | proof | source | tool | human | shadow_eval
  producer
  command_or_source
  digest
  timestamp
  supports
  refutes

Objection:
  id
  target_claim
  severity
  description
  proposed_resolution
  status
```

This would make weak-model reasoning inspectable, mergeable, and memory-safe.

## C. Memory curator gate

Permanent memory should require:

* promotion receipt;
* evidence bundle;
* contradiction scan;
* redaction;
* provenance hash;
* curator summary;
* rollback path.

This aligns with ZYAL’s existing memory/provenance direction and Cogcore’s deterministic ledger approach. ([GitHub][2])

## D. Adaptive lane allocator

Create a lane allocator that decides how many workers to spawn based on:

* estimated task difficulty;
* model capability;
* verification availability;
* disagreement level;
* risk level;
* remaining budget;
* negative-memory hits.

Rule of thumb:

```text
weaker model + high uncertainty + cheap verification => more lanes
stronger model + low uncertainty + expensive verification => fewer lanes
high risk + no verifier => human gate or conservative answer
```

## E. Separate judge model when available

ZYAL already points toward critics on different providers/families and confidence caps. ([GitHub][1]) This is important because same-model judging can share the same blind spots as same-model solving.

When different providers are unavailable, create diversity through:

* different prompts;
* different context packs;
* different decomposition strategies;
* delayed critique;
* hidden tests;
* independent reducer heuristics.

## F. Reasoning benchmark suite

Create a benchmark harness similar to the memory-benchmark/AutoResearch approach, but for reasoning workflows.

Evaluate:

* multi-hop factual reasoning;
* code diagnosis;
* math/proof sketches;
* planning under constraints;
* contradiction detection;
* memory reuse;
* false-memory resistance;
* adversarial prompt injection;
* source-grounded synthesis;
* long-horizon task repair.

The Memory System Levelup notes already emphasize verification gates, deterministic tests, benchmark targets, shadow suites, trusted-core gates, and not arming unsafe chase-daemon behavior until clean-source and benchmark gates pass. ([GitHub][8])

# The actual operating philosophy

The winning formula is:

```text
Power = decomposition
      × diversity
      × independent search
      × adversarial critique
      × external verification
      × safe memory
      × host-owned control
```

A cluster of weak models will not reliably become strong by talking more. It becomes strong when ZYAL turns them into a **bounded scientific process**:

* propose;
* isolate;
* test;
* criticize;
* synthesize;
* remember;
* repeat.

That is the design I would push hardest: **ZYAL as a reasoning operating system, not an agent prompt format.**
